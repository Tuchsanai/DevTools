🎯 ภาพรวม & ทำไมต้อง Docker

เป้าหมายของเครื่องมือชุดนี้ และเหตุผลที่เราใช้ Docker

วัตถุประสงค์: เรียนรู้ เครื่องมือพัฒนาซอฟต์แวร์ (Software Development Tools) บนสภาพแวดล้อม Linux (Ubuntu 24.04) ที่มี Docker, Git, Python 3, Node.js ฯลฯ ติดตั้งมาให้พร้อมและเหมือนกันทุกคน

🔧

ความท้าทาย: นักศึกษาส่วนใหญ่ใช้ Windows เป็นหลัก ซึ่งต่างจากเซิร์ฟเวอร์/สภาพแวดล้อมจริงที่เป็น Linux

💡

วิธีแก้: ใช้ Docker Container จำลองสภาพแวดล้อม Linux ขึ้นมาบน Windows แล้วทำงานข้างในนั้น

ทำไมต้อง Docker?

⚡

สะดวก

เรียกใช้ Linux บน Windows ได้ทันที ไม่ต้องลง dual-boot หรือ VM ที่ยุ่งยาก

🧩

สม่ำเสมอ

ทุกคนใช้สภาพแวดล้อมเดียวกันเป๊ะ ลดปัญหา “เครื่องฉันรันได้ เครื่องเธอรันไม่ได้”

📦

ลดปัญหา

ไม่ต้องไล่ติดตั้ง dependencies แยกบน Windows — มากับ image เรียบร้อย

🛡️

ปลอดภัย

ทำงานแยกใน container ไม่กระทบระบบ Windows ของคุณ ลบทิ้งแล้วเครื่องสะอาดเหมือนเดิม

🧰 สิ่งที่ต้องเตรียม

ก่อนเริ่มทำตามคู่มือ ตรวจสอบว่าพร้อมตามนี้

  • เครื่อง Windows 10/11 ที่มีสิทธิ์ติดตั้งโปรแกรม (Administrator)
  • พื้นที่ดิสก์ว่างพอสมควร (image มีขนาดหลาย GB เพราะรวม Docker, Node.js และเครื่องมือไว้แล้ว)
  • การเชื่อมต่ออินเทอร์เน็ต (สำหรับดาวน์โหลด Docker Desktop และดึง image ครั้งแรก)
  • VS Code (Visual Studio Code) — ใช้จัดการ container แบบคลิกได้ง่าย
📝

Docker Desktop จะเปิดใช้ WSL2 backend ให้โดยอัตโนมัติ ถ้าระบบยังไม่มี อาจถูกขอให้ติดตั้ง/อัปเดต WSL2 และรีสตาร์ตเครื่องหนึ่งครั้ง

1 ติดตั้ง Docker Desktop

ติดตั้งโปรแกรมหลักที่ใช้รัน container บน Windows

  1. ดาวน์โหลด Docker Desktop จากเว็บทางการ https://www.docker.com/products/docker-desktop แล้วกดปุ่ม Download Docker Desktop (เลือกเวอร์ชันสำหรับ Windows)
  2. ติดตั้งแล้วรีบูตเครื่อง ทำตามตัวติดตั้งจนเสร็จ แล้ว restart คอมพิวเตอร์หนึ่งครั้ง
  3. เปิด Docker Desktop รอจนแถบสถานะมุมล่างซ้ายขึ้นว่า Engine running (สีเขียว)
  4. ทดสอบจาก PowerShell หรือ Command Prompt พิมพ์คำสั่งด้านล่าง ควรเห็นเลขเวอร์ชันตอบกลับ
PowerShell / Command Prompt 
docker --version
⚠️

ถ้าขึ้นว่า “docker : command not found” หรือ “not recognized”: มักเป็นเพราะ Docker Desktop ยังไม่เปิด หรือยังไม่ได้รีสตาร์ตหลังติดตั้ง → เปิด Docker Desktop ให้ขึ้น Engine running ก่อน แล้วเปิดหน้าต่างเทอร์มินัลใหม่อีกครั้ง

2 ตั้งค่า Docker ใน VS Code

ติดตั้ง Docker extension เพื่อจัดการ container ด้วยการคลิก ไม่ต้องจำคำสั่งเยอะ

  1. เปิดแถบ Extensions ใน VS Code (ไอคอนรูปสี่เหลี่ยมที่ Activity Bar ด้านซ้าย หรือกด Ctrl+Shift+X)
  2. ค้นหา “docker” เลือกตัวที่เป็นของ Microsoft (มีเครื่องหมายยืนยัน ✔)
  3. กด Install รอจนติดตั้งเสร็จ (identifier: ms-azuretools.vscode-docker)
  4. ตรวจผล จะมีไอคอน/แถบ CONTAINERS โผล่ขึ้นมาที่ Activity Bar ด้านซ้าย
  5. ทดสอบการเชื่อมต่อ เปิด Terminal ใน VS Code (Ctrl+Shift+`) แล้วพิมพ์ docker --version
VS Code Terminal 
docker --version
💡

ทำไมต้องใช้ใน VS Code? เพราะ extension นี้ให้เรา ดู / start / stop / login เข้า container ได้ด้วยการคลิกขวา และยังเปิดโค้ดข้างใน container มาแก้ได้โดยตรง (จะใช้ในขั้นที่ 4–5)

3 ดึงและรัน Container

นำ image ของวิชามารันเป็น container แล้วเข้าใช้งานผ่านเชลล์ / SSH

📦 ข้างใน image มีอะไร?

image tuchsanai/devtools:2569_1 สร้างจาก Ubuntu 24.04 แล้วติดตั้งชุดเครื่องมือพัฒนาซอฟต์แวร์ไว้ให้พร้อม — Docker-in-Docker, Python 3, Node.js 22 (LTS), Git และ SSH (ตัวอย่าง Dockerfile แบบย่อ):

▶️ คำสั่งรัน Container

PowerShell / VS Code Terminal 
docker run -d --name devtools --privileged -p 2222:22 tuchsanai/devtools:2569_1
ส่วนของคำสั่งความหมาย
-dรันแบบ detached (เบื้องหลัง) — ไม่ค้างหน้าจอเทอร์มินัล
--name devtoolsตั้งชื่อ container ให้เรียกใช้ง่าย (แทนการจำ Container ID)
--privilegedจำเป็นสำหรับ Docker-in-Docker — ให้ dockerd ภายใน container ทำงานได้
-p 2222:22แมปพอร์ต 22 (SSH) ใน container ออกมาที่พอร์ต 2222 บนเครื่อง Windows
tuchsanai/devtools:2569_1ชื่อ image ของวิชา (tag 2569_1)
  1. รันคำสั่งด้านบน ครั้งแรก Docker จะ pull (ดาวน์โหลด) image จาก Docker Hub — ใช้เวลาสักครู่ตามความเร็วเน็ต
  2. ตรวจว่า container กำลังรัน ดูได้ทั้งใน VS Code (แถบ CONTAINERS) และใน Docker Desktop
  3. เข้าใช้งาน container ผ่านเชลล์ docker exec -it devtools bash หรือ SSH ssh root@localhost -p 2222 (ดูรายละเอียดในขั้นที่ 4–5)
🔑

การเข้าใช้งาน: image นี้ไม่มี JupyterLab — เข้าผ่าน เชลล์หรือ SSH แทน · เชลล์: docker exec -it devtools bash · SSH: ssh root@localhost -p 2222 (container รันเป็นผู้ใช้ root โฟลเดอร์ทำงาน /workspace)

⚠️

พอร์ต 2222 ถูกใช้อยู่แล้ว? (ขึ้น error port is already allocated) ให้เปลี่ยนพอร์ตฝั่งซ้าย เช่น -p 2223:22 แล้ว SSH ด้วย ssh root@localhost -p 2223 แทน

4 เข้าใช้งาน Container — วิธีที่ 1: Attach Shell

เปิดเทอร์มินัล (เชลล์) เข้าไปทำงานข้างใน container โดยตรง

  1. คลิกขวาที่ container ในแถบ CONTAINERS ของ VS Code (ใต้ Individual Containers)
  2. เลือก “Attach Shell” จากเมนูที่เด้งขึ้นมา
  3. จะได้เทอร์มินัล bash ภายใน container ลองทดสอบเครื่องมือที่ติดตั้งมาให้
เทอร์มินัลภายใน container 
root@devtools:/workspace# python --version
root@devtools:/workspace# node --version
root@devtools:/workspace# docker --version
root@devtools:/workspace# git --version
🧠

เหมาะกับ: รันคำสั่งเร็ว ๆ, ติดตั้ง package เพิ่ม, จัดการไฟล์ภายใน container — เทียบเท่ากับการพิมพ์ docker exec -it devtools bash ในเทอร์มินัลปกติ

5 เข้าใช้งาน Container — วิธีที่ 2: Attach VS Code

เปิด VS Code เชื่อมต่อตรงเข้า container เพื่อเขียน/แก้โค้ดเหมือนทำในเครื่อง

  1. คลิกขวาที่ container ในแถบ CONTAINERS เหมือนเดิม
  2. เลือก “Attach Visual Studio Code”
  3. VS Code จะเปิดหน้าต่างใหม่ที่ต่ออยู่กับ container สังเกตป้าย Container … ที่มุมล่างซ้าย
  4. กด Open Folder เลือก /workspace เพื่อเริ่มเขียนโค้ด/รันโปรเจกต์ภายใน container

⚖️ เลือกวิธีไหนดี?

วิธีที่ 1 · Attach Shell

  • ได้ เทอร์มินัล เข้า container ทันที
  • เบา เร็ว เหมาะรันคำสั่งสั้น ๆ
  • ติดตั้ง package, เช็กเวอร์ชัน, จัดการไฟล์

วิธีที่ 2 · Attach VS Code

  • ได้ โปรแกรมแก้โค้ดเต็มรูปแบบ ใน container
  • เปิดโฟลเดอร์, แก้ไฟล์, ใช้ extensions, debug
  • เหมาะเขียนโค้ดยาว ๆ และรันโปรเจกต์ (Next.js / React / Python ฯลฯ)
📝

ทั้งสองวิธีเข้า container เดียวกัน — เลือกตามงาน: อยากพิมพ์คำสั่งไว ๆ ใช้วิธีที่ 1, อยากนั่งเขียนโค้ดจริงจังใช้วิธีที่ 2 (และยังเข้าผ่าน SSH ssh root@localhost -p 2222 ควบคู่ได้ตลอด)
