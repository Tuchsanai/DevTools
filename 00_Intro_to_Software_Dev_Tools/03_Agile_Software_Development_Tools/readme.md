## 1. Agile คืออะไร?

> 💡 **Agile** ไม่ใช่เครื่องมือ ไม่ใช่ขั้นตอน แต่คือ **แนวคิด (Mindset)** ในการพัฒนาซอฟต์แวร์ที่เน้นความยืดหยุ่น การส่งมอบคุณค่าอย่างต่อเนื่อง และการทำงานร่วมกันของทีม

### 1.1 ปัญหาของวิธีเดิม — ทำไมถึงต้องมี Agile?

```
  ❌ กรณีศึกษา: โปรเจกต์ระบบ ERP ด้วยวิธี Waterfall แบบดั้งเดิม

  มกราคม       มีนาคม        มิถุนายน       ตุลาคม        ธันวาคม
  ───────────  ────────────  ─────────────  ────────────  ──────────
  วิเคราะห์    ออกแบบ        พัฒนา          ทดสอบ         ส่งมอบ
  Requirements Architecture  (4 เดือน)      UAT           ❌ FAIL
  (2 เดือน)   (3 เดือน)
       │
       └─► ลูกค้าบอก "ต้องการระบบ Payroll"
              │
              ▼
       ตุลาคม: ลูกค้าเปลี่ยนใจ → "เพิ่ม Module HR ด้วย"
              │
              ▼
       ธันวาคม: ส่งมอบระบบที่ลูกค้าไม่ต้องการอีกแล้ว
               ใช้เงิน 12 ล้านบาท ใช้เวลา 12 เดือน
               แต่ Business เปลี่ยนไปแล้วทั้งหมด

  ──────────────────────────────────────────────────────────────
  สรุปปัญหาหลักของ Waterfall:

  • รับ Feedback ช้า → รู้ว่าพลาดตอน Deploy แล้วเท่านั้น
  • เปลี่ยนแปลง Requirement ยาก → Cost สูงมากถ้าเปลี่ยนระหว่างทาง
  • ส่งมอบช้า → ลูกค้าเห็นผลลัพธ์จริงครั้งแรกในเดือนที่ 12
  • ความเสี่ยงสูง → ลงทุนเต็มที่ก่อน แล้วค่อยรู้ว่าถูกผิดทีหลัง
```

## 4. Scrum Framework

> 💡 **Scrum** คือ Lightweight Framework ที่ช่วยให้คน ทีม และองค์กรสร้างคุณค่าผ่านการ Adapt กับปัญหาที่ซับซ้อน โดยมีโครงสร้างที่ชัดเจนประกอบด้วย Roles, Events, และ Artifacts

### 4.1 Scrum Theory — 3 เสาหลัก

```
╔══════════════════════════════════════════════════════════════════╗
║               🏛️ SCRUM EMPIRICISM — 3 PILLARS                    ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  TRANSPARENCY (ความโปร่งใส)                                      ║
║  → งาน, ความก้าวหน้า, ปัญหา ต้องมองเห็นได้โดยทุกคนในทีม         ║
║  → ไม่มีการซ่อน Progress หรือ Block ที่แท้จริง                   ║
║  → ตัวอย่าง: Scrum Board ที่ทุกคนเห็น Status ได้ตลอดเวลา        ║
║                                                                  ║
║  INSPECTION (การตรวจสอบ)                                         ║
║  → ตรวจสอบ Artifacts และความก้าวหน้าสม่ำเสมอเพื่อตรวจจับ        ║
║    ความเบี่ยงเบนจาก Goal                                         ║
║  → ตัวอย่าง: Sprint Review นำเสนองานให้ Stakeholders ดู          ║
║                                                                  ║
║  ADAPTATION (การปรับตัว)                                         ║
║  → เมื่อตรวจสอบแล้วพบว่าเบี่ยงเบน ต้องปรับโดยเร็วที่สุด          ║
║  → ตัวอย่าง: Sprint Retrospective ปรับกระบวนการทำงาน             ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

### 4.2 Scrum Values — 5 ค่านิยม

```
                      💎 SCRUM VALUES

             COMMITMENT          COURAGE
             (ความมุ่งมั่น)     (ความกล้า)
                    │                │
                    └───────┬────────┘
                            │
                         FOCUS
                        (สมาธิ)
                            │
                    ┌───────┴────────┐
                    │                │
               OPENNESS           RESPECT
              (ความเปิดเผย)    (ความเคารพ)

  • COMMITMENT  — ทีมมุ่งมั่นต่อเป้าหมาย ไม่ใช่แค่ทำพอผ่าน
  • COURAGE     — กล้าทำสิ่งยาก กล้าพูดความจริง กล้าขอความช่วยเหลือ
  • FOCUS       — โฟกัสที่ Sprint Goal ไม่วอกแวก
  • OPENNESS    — เปิดเผยต่องาน ความท้าทาย และความเรียนรู้
  • RESPECT     — สมาชิกในทีมเคารพศักดิ์ศรีและความสามารถกัน
```

### 4.3 Scrum Roles — 3 บทบาทหลัก

```
╔══════════════════════════════════════════════════════════════════╗
║                    👥 SCRUM TEAM ROLES                           ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  🎯 PRODUCT OWNER (1 คน)                                         ║
║  ─────────────────────────────────────────────────              ║
║  หน้าที่: เป็นตัวแทนของลูกค้าและ Business ในทีม                  ║
║                                                                  ║
║  ความรับผิดชอบหลัก:                                              ║
║  • สร้างและดูแล Product Backlog                                  ║
║  • กำหนดลำดับความสำคัญ (Priority) ของ Backlog Items             ║
║  • ตัดสินใจว่า Sprint นี้จะทำอะไร                                ║
║  • Accept หรือ Reject งานที่ทีมทำเสร็จ                          ║
║  • สื่อสาร Product Vision กับทีม                                 ║
║                                                                  ║
║  ❌ สิ่งที่ PO ไม่ควรทำ:                                         ║
║  • สั่งให้ Developer ทำงานโดยตรง (ต้องผ่าน Backlog)             ║
║  • มีหลายคนในบทบาทนี้พร้อมกัน (ต้องมี 1 เสียงเท่านั้น)           ║
║                                                                  ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  🛡️ SCRUM MASTER (1 คน)                                          ║
║  ─────────────────────────────────────────────────              ║
║  หน้าที่: Servant Leader ที่ช่วยให้ทีม และองค์กร ใช้ Scrum      ║
║           ได้อย่างมีประสิทธิภาพ                                   ║
║                                                                  ║
║  ความรับผิดชอบหลัก:                                              ║
║  • Facilitate Scrum Events ทุกอย่าง                              ║
║  • Remove Impediments (สิ่งที่ขัดขวางทีม)                        ║
║  • Coach ทีมเรื่อง Scrum และ Agile Practices                     ║
║  • ปกป้องทีมจาก External Interruptions                           ║
║  • ช่วย Organization เข้าใจและ Adopt Scrum                       ║
║                                                                  ║
║  💡 SM ≠ Project Manager                                         ║
║     PM: ควบคุมทีม, จัดการทรัพยากร, รายงาน Management            ║
║     SM: รับใช้ทีม, ขจัดอุปสรรค, Coach ไม่ใช่ Command            ║
║                                                                  ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  👨‍💻 DEVELOPERS (3-9 คน)                                         ║
║  ─────────────────────────────────────────────────              ║
║  หน้าที่: สร้าง Increment ที่มีคุณค่าในแต่ละ Sprint              ║
║                                                                  ║
║  ความรับผิดชอบหลัก:                                              ║
║  • วางแผนงาน Sprint ใน Sprint Planning                           ║
║  • สร้าง Sprint Backlog                                          ║
║  • Adapt Plan ทุกวันใน Daily Scrum                               ║
║  • สร้าง Increment ที่ผ่าน Definition of Done                    ║
║  • เป็น Cross-functional: Frontend, Backend, QA, DevOps         ║
║                                                                  ║
║  💡 ใน Scrum ไม่มี Sub-team หรือ Hierarchy                       ║
║     ทุกคนคือ Developer ที่รับผิดชอบต่อผลลัพธ์ร่วมกัน             ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

### 4.4 Scrum Events — 5 Event หลัก

```
  📅 SCRUM EVENTS ทั้งหมดอยู่ใน Sprint

  ┌───────────────────────────────────────────────────────────────┐
  │                         SPRINT (1-4 สัปดาห์)                  │
  │                                                               │
  │  ┌──────────┐                                    ┌─────────┐  │
  │  │ SPRINT   │                                    │ SPRINT  │  │
  │  │ PLANNING │◄── Day 1                Day N ────►│ REVIEW  │  │
  │  └──────────┘                                    └────┬────┘  │
  │       │                                               │       │
  │       ▼                                               ▼       │
  │  Sprint Backlog                              ┌────────────────┐│
  │  สร้างขึ้นมา                                 │   SPRINT       ││
  │                                             │ RETROSPECTIVE  ││
  │  ┌──────────────────────────────────────┐   └────────────────┘│
  │  │    DAILY SCRUM (ทุกวัน 15 นาที)       │                    │
  │  │    เช้า 09:00 — Stand Up Meeting     │                    │
  │  └──────────────────────────────────────┘                    │
  │                                                               │
  └───────────────────────────────────────────────────────────────┘
              + BACKLOG REFINEMENT (ไม่ใช่ Official Event
                แต่ทำระหว่าง Sprint เพื่อเตรียม Backlog)
```

#### 4.4.1 Sprint

```
  🏃 SPRINT — หัวใจของ Scrum

  ✅ Sprint คือ:
  • Time-box 1-4 สัปดาห์ (ส่วนใหญ่ 2 สัปดาห์)
  • Container ของ Scrum Events ทั้งหมด
  • ผลลัพธ์ต้องเป็น "Done Increment" เสมอ
  • ทุก Sprint มี Sprint Goal ที่ชัดเจน 1 อย่าง

  ❌ ห้ามทำระหว่าง Sprint:
  • เปลี่ยน Sprint Goal กลางคัน
  • ลด Quality Standards
  • เพิ่มงานใน Sprint Backlog โดยไม่ตกลงกับทีม

  💡 ถ้า Sprint Goal หมดความหมายแล้ว
     → Product Owner สามารถยกเลิก Sprint ได้
     → แต่หายากมากในทางปฏิบัติ
```

#### 4.4.2 Sprint Planning

```
╔══════════════════════════════════════════════════════════════════╗
║  📋 SPRINT PLANNING — เกิดขึ้น: ต้นของแต่ละ Sprint              ║
║                        เวลา: สูงสุด 8 ชั่วโมง (สำหรับ 1 เดือน)  ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  ตอบ 3 คำถามสำคัญ:                                               ║
║                                                                  ║
║  1️⃣  "WHY is this Sprint valuable?"                              ║
║      Product Owner เสนอว่า Sprint นี้ควรสร้างคุณค่าอะไร         ║
║      → ทีมกำหนด Sprint Goal ร่วมกัน                             ║
║                                                                  ║
║  2️⃣  "WHAT can be Done this Sprint?"                             ║
║      Developers เลือก Backlog Items จาก Product Backlog         ║
║      → Negotiate กับ PO ว่าทำได้แค่ไหน                          ║
║                                                                  ║
║  3️⃣  "HOW will the chosen work get done?"                        ║
║      Developers แตก Backlog Items เป็น Tasks ย่อย               ║
║      → ประมาณเวลาแต่ละ Task                                     ║
║      → สร้าง Sprint Backlog                                     ║
║                                                                  ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  ตัวอย่าง Sprint Planning:                                       ║
║                                                                  ║
║  Sprint Goal: "ผู้ใช้สามารถสมัครสมาชิกและล็อกอินได้"             ║
║                                                                  ║
║  Backlog Items ที่เลือก:                                         ║
║  ☐ US-01: สมัครสมาชิกด้วย Email     (5 points)                  ║
║  ☐ US-02: ล็อกอินด้วย Email/Password (3 points)                 ║
║  ☐ US-03: Reset Password ผ่าน Email  (3 points)                 ║
║                                                                  ║
║  Tasks (เฉพาะ US-01):                                            ║
║  ☐ Task: สร้าง Register API Endpoint      (4h) @Alice           ║
║  ☐ Task: เขียน Unit Test สำหรับ Register  (2h) @Alice           ║
║  ☐ Task: สร้าง Register Form UI          (3h) @Bob              ║
║  ☐ Task: Validate Form Input (Frontend)  (2h) @Bob              ║
║  ☐ Task: Email Verification Flow         (4h) @Charlie          ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

#### 4.4.3 Daily Scrum

```
  ⏰ DAILY SCRUM — ประชุมประจำวัน

  ┌──────────────────────────────────────────────────────────────┐
  │  เวลา: ทุกวัน 15 นาที (เวลาและสถานที่เดิมทุกวัน)              │
  │  ผู้เข้าร่วม: Developers เท่านั้น (SM เข้าได้ถ้าทีมต้องการ)  │
  │  รูปแบบ: Stand-up ยืนคุย (ป้องกันการนั่งหลับ!)               │
  └──────────────────────────────────────────────────────────────┘

  ❌ แบบเก่า (ไม่แนะนำ):
  "เมื่อวานทำ... วันนี้จะทำ... มี Block ไหม?"
  → กลายเป็น Status Report ให้ Manager ฟัง

  ✅ แบบใหม่ (Scrum Guide 2020):
  ทีมพูดคุยเพื่อ Inspect Progress สู่ Sprint Goal
  และปรับ Sprint Backlog ตามความจำเป็น

  ตัวอย่างการ Stand-up ที่ดี:
  ┌─────────────────────────────────────────────────────────────┐
  │  Alice: "Register API เสร็จแล้ว อยู่ระหว่าง Unit Test      │
  │          วันนี้จะ merge ให้เสร็จ แล้วช่วย Bob ส่วน UI"      │
  │                                                             │
  │  Bob:   "Form UI ทำได้ 70% แล้ว แต่ติดเรื่อง Design        │
  │          ที่ Figma ยังไม่ clear ขอเวลา 5 นาทีหลัง Daily     │
  │          คุยกับ Alice ได้ไหม?"                              │
  │                                                             │
  │  Alice: "ได้เลย"                                            │
  └─────────────────────────────────────────────────────────────┘

  💡 ประเด็นที่ลึกกว่า ออกไปคุยต่อนอก Daily Scrum
     ไม่ใช้เวลา 15 นาทีแก้ปัญหาเทคนิค
```

#### 4.4.4 Sprint Review

```
  🎭 SPRINT REVIEW — Demo งานให้ Stakeholders

  ┌──────────────────────────────────────────────────────────────┐
  │  เวลา: ท้าย Sprint, สูงสุด 4 ชั่วโมง (สำหรับ 1 เดือน)        │
  │  ผู้เข้าร่วม: Scrum Team + Stakeholders + ลูกค้า             │
  │  จุดประสงค์: Inspect Increment, Adapt Product Backlog        │
  └──────────────────────────────────────────────────────────────┘

  Sprint Review Flow:
  ─────────────────
  1. Product Owner อธิบาย "Done" Items
  2. Developers Demo Working Software (ไม่ใช่ Slides!)
  3. Stakeholders ให้ Feedback
  4. Product Owner ปรับ Product Backlog ตาม Feedback
  5. วางแผน Sprint ต่อไปร่วมกัน

  ❌ Sprint Review ≠ Approval Meeting
     ไม่ใช่การขอ "ผ่าน" จาก Management
     แต่คือการ Collaborate เพื่อให้ Product ดีขึ้น
```

#### 4.4.5 Sprint Retrospective

```
╔══════════════════════════════════════════════════════════════════╗
║  🔍 SPRINT RETROSPECTIVE — ปรับปรุงวิธีทำงานของทีม              ║
╠══════════════════════════════════════════════════════════════════╣
║  เวลา: หลัง Sprint Review, สูงสุด 3 ชั่วโมง                    ║
║  ผู้เข้าร่วม: Scrum Team เท่านั้น (ไม่มี Stakeholders)          ║
║  จุดประสงค์: ปรับปรุง Process ทีม ไม่ใช่ Product               ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  3 คำถามสำคัญ:                                                   ║
║                                                                  ║
║  ✅ WHAT WENT WELL? (สิ่งที่ดีแล้ว ควรทำต่อ)                    ║
║     "Code Review ทำให้ Bug ลดลงมาก"                            ║
║     "Daily Stand-up ตรงเวลาทุกวัน"                              ║
║                                                                  ║
║  ⚠️ WHAT COULD BE IMPROVED? (สิ่งที่ต้องปรับปรุง)               ║
║     "Estimation ผิดพลาดบ่อย Tasks ซับซ้อนเกินคาด"              ║
║     "Communication กับ PO ไม่ทันเวลา"                           ║
║                                                                  ║
║  🎯 ACTION ITEMS (สิ่งที่จะทำใน Sprint ต่อไป)                   ║
║     "จะลอง Planning Poker ในการ Estimate"                       ║
║     "PO จะ Available ทุก Wednesday 14:00-15:00"                 ║
║                                                                  ║
╠══════════════════════════════════════════════════════════════════╣
║  Retro Format ยอดนิยม:                                           ║
║                                                                  ║
║  Start / Stop / Continue                                         ║
║  ┌────────────┬────────────┬────────────┐                        ║
║  │   START    │    STOP    │  CONTINUE  │                        ║
║  │            │            │            │                        ║
║  │ สิ่งที่ยัง  │ สิ่งที่ทำ  │ สิ่งที่ดี  │                        ║
║  │ ไม่ได้ทำ   │ อยู่แต่    │ อยู่แล้ว  │                        ║
║  │ แต่ควรทำ   │ ควรหยุด    │ ทำต่อไป   │                        ║
║  └────────────┴────────────┴────────────┘                        ║
║                                                                  ║
```

### 4.5 Scrum Artifacts — 3 สิ่งที่จับต้องได้

```
  📦 SCRUM ARTIFACTS — ข้อมูลที่สำคัญต่อความโปร่งใส

  ARTIFACT 1: Product Backlog
  ──────────────────────────
  • Ordered List ของทุกสิ่งที่ Product ต้องการ
  • PO เป็นเจ้าของและรับผิดชอบ
  • ปรับเปลี่ยนได้ตลอดเวลาตาม Feedback
  • มี Commitment เรียกว่า "Product Goal"

  ตัวอย่าง Product Backlog:
  ┌────┬─────────────────────────────────────────┬────────┬──────┐
  │ #  │ Story                                   │ Points │ Prio │
  ├────┼─────────────────────────────────────────┼────────┼──────┤
  │ 1  │ ผู้ใช้สามารถ Login ด้วย Email             │   3    │  🔴  │
  │ 2  │ ผู้ใช้สามารถ Reset Password               │   3    │  🔴  │
  │ 3  │ ผู้ใช้สามารถดู Order History              │   5    │  🟠  │
  │ 4  │ แสดง Notification แบบ Real-time           │   8    │  🟡  │
  │ 5  │ ระบบรองรับหลายภาษา (Localization)        │  13    │  🟢  │
  └────┴─────────────────────────────────────────┴────────┴──────┘
  🔴 Must Have  🟠 Should Have  🟡 Could Have  🟢 Won't Have (now)

  ──────────────────────────────────────────────────────────────

  ARTIFACT 2: Sprint Backlog
  ─────────────────────────
  • Backlog Items ที่เลือกมาทำใน Sprint นี้ + Plan ว่าจะทำอย่างไร
  • Developers เป็นเจ้าของ
  • อัปเดตทุกวันตาม Daily Scrum
  • มี Commitment เรียกว่า "Sprint Goal"

  ──────────────────────────────────────────────────────────────

  ARTIFACT 3: Increment
  ─────────────────────
  • ผลงานที่ทำได้ใน Sprint + Increments ของ Sprint ก่อนหน้า
  • ต้องผ่าน Definition of Done (DoD)
  • ต้องอยู่ในสภาพที่ Usable และ Potentially Releasable
  • มี Commitment เรียกว่า "Definition of Done"
```

### 4.6 Definition of Done (DoD)

```
╔══════════════════════════════════════════════════════════════════╗
║  ✅ DEFINITION OF DONE — มาตรฐานคุณภาพของทีม                    ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  "Done" ไม่ได้แปลว่า "โค้ดเขียนเสร็จ"                           ║
║  "Done" หมายถึงผ่านเกณฑ์ทุกข้อที่ทีมกำหนดไว้                    ║
║                                                                  ║
║  ตัวอย่าง Definition of Done ของทีมพัฒนาทั่วไป:                  ║
║                                                                  ║
║  ☑️ โค้ดถูก Review โดยสมาชิกทีมอย่างน้อย 1 คน                   ║
║  ☑️ Unit Test เขียนครบและ Pass ทั้งหมด (Coverage > 80%)          ║
║  ☑️ Integration Test Pass                                        ║
║  ☑️ โค้ด Merge เข้า Main Branch แล้ว                             ║
║  ☑️ Deploy ขึ้น Staging Environment สำเร็จ                       ║
║  ☑️ QA ทดสอบและ Approve บน Staging                               ║
║  ☑️ Documentation อัปเดตแล้ว (ถ้ามีการเปลี่ยนแปลง API)           ║
║  ☑️ Performance ไม่แย่กว่า Baseline ที่กำหนด                     ║
║  ☑️ ไม่มี Known Security Vulnerability ที่ยังไม่แก้               ║
║                                                                  ║
║  💡 DoD เป็นสิ่งที่ทีมกำหนดร่วมกัน และปรับได้ใน Retrospective   ║
║     แต่ละ Story จะมี "Acceptance Criteria" เพิ่มเติมอีกด้วย      ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

### 4.7 User Stories — วิธีเขียน Backlog Items

```
  📝 USER STORY FORMAT

  "As a [type of user],
   I want [some goal],
   so that [some reason]"

  ตัวอย่าง:
  ─────────
  As a registered user,
  I want to reset my password via email,
  so that I can regain access to my account when I forget my password.

  Acceptance Criteria (เกณฑ์รับงาน):
  ☐ เมื่อกด "Forgot Password" จะมีหน้าให้กรอก Email
  ☐ ระบบส่ง Email พร้อม Link รีเซ็ตภายใน 1 นาที
  ☐ Link มีอายุ 24 ชั่วโมงเท่านั้น
  ☐ หลังรีเซ็ตสำเร็จ Link เก่าใช้งานไม่ได้อีก
  ☐ แสดง Error message ที่เหมาะสมเมื่อ Email ไม่มีในระบบ

  ────────────────────────────────────────────────────────────
  💡 INVEST Criteria — User Story ที่ดีต้องเป็น:

  I — Independent   แต่ละ Story เป็นอิสระต่อกันได้
  N — Negotiable    ปรับเปลี่ยนได้ตาม Conversation
  V — Valuable      สร้างคุณค่าให้ผู้ใช้หรือ Business
  E — Estimable     ประมาณขนาดงานได้
  S — Small         เล็กพอทำเสร็จใน 1 Sprint
  T — Testable      ทดสอบได้ว่าทำเสร็จแล้วจริง
```

### 4.8 Story Points & Estimation

```
  🃏 PLANNING POKER — วิธี Estimate ที่นิยมใน Scrum

  ทำไมต้อง Story Points ไม่ใช่ ชั่วโมง?
  ─────────────────────────────────────
  ชั่วโมง: "Login Page ทำ 4 ชั่วโมง"
  → แต่ละคนทำได้ไม่เท่ากัน, ขึ้นอยู่กับประสบการณ์

  Story Points: "Login Page = 3 Points" (ความซับซ้อนสัมพัทธ์)
  → ถ้า Register Page = 5 Points แสดงว่า Register ยากกว่า Login ~67%
  → ทีมเรียนรู้ Velocity ของตัวเองตามเวลา

  ────────────────────────────────────────────────────────────
  วิธีเล่น Planning Poker:

  1. PO อ่าน User Story ให้ทีมฟัง
  2. ทีมถามคำถามเพื่อทำความเข้าใจ
  3. ทุกคนเลือก Card พร้อมกัน (Fibonacci: 1,2,3,5,8,13,21,?)
  4. เปิด Card พร้อมกัน
  5. ถ้าค่าใกล้กัน → ใช้ค่ากลาง
  6. ถ้าค่าต่างกันมาก → คนที่สูงสุดและต่ำสุดอธิบาย
  7. Vote ใหม่จนได้ Consensus

  ┌───┬───┬───┬───┬───┬────┬────┬────┬────┐
  │ 1 │ 2 │ 3 │ 5 │ 8 │ 13 │ 21 │ 34 │  ? │
  └───┴───┴───┴───┴───┴────┴────┴────┴────┘
  ? = "ไม่รู้ ต้องการข้อมูลเพิ่มก่อน Estimate"
```

## 5. Kanban Method

> 💡 **Kanban** มาจากภาษาญี่ปุ่น แปลว่า "กระดานสัญญาณ" มีต้นกำเนิดจากระบบการผลิตของ Toyota ในทศวรรษ 1950 ก่อนถูกนำมาปรับใช้ในงาน Knowledge Work และการพัฒนาซอฟต์แวร์

### 5.1 Kanban Core Principles — 4 หลักการ

```
╔══════════════════════════════════════════════════════════════════╗
║           🎌 KANBAN CORE PRINCIPLES                              ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  1. START WITH WHAT YOU DO NOW (เริ่มจากสิ่งที่ทำอยู่)           ║
║     → ไม่ต้องเปลี่ยนกระบวนการทั้งหมดทันที                        ║
║     → Visualize งานที่ทำอยู่แล้วก่อน                             ║
║     → Kanban ปรับตัวเข้ากับ Existing Process                    ║
║                                                                  ║
║  2. AGREE TO PURSUE INCREMENTAL, EVOLUTIONARY CHANGE            ║
║     → เปลี่ยนแปลงทีละเล็กทีละน้อย ไม่ใช่ Big Bang                ║
║     → ทดลอง → วัดผล → ปรับปรุง                                   ║
║                                                                  ║
║  3. RESPECT CURRENT PROCESS, ROLES, RESPONSIBILITIES            ║
║     → ไม่บังคับเปลี่ยน Title หรือ Role ทันที                     ║
║     → เคารพสิ่งที่ทำงานได้ดีอยู่แล้ว                             ║
║                                                                  ║
║  4. ENCOURAGE ACTS OF LEADERSHIP AT ALL LEVELS                  ║
║     → ทุกคนในทีมมีส่วนร่วมในการปรับปรุง                          ║
║     → ไม่ใช่แค่ผู้บริหารที่ตัดสินใจ                               ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

### 5.4 WIP Limit — กุญแจสำคัญของ Kanban

```
╔══════════════════════════════════════════════════════════════════╗
║     🚦 WIP LIMIT (Work In Progress Limit) — ทำไมถึงสำคัญ?       ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  🧠 Little's Law: Cycle Time = WIP ÷ Throughput                  ║
║                                                                  ║
║  ลดงานที่ทำพร้อมกัน → Cycle Time สั้นลง → ส่งมอบเร็วขึ้น         ║
║                                                                  ╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  ❌ ไม่มี WIP Limit (ทำหลายอย่างพร้อมกัน):                       ║
║                                                                  ║
║  Alice: กำลังทำ Feature A (30%), B (20%), C (10%)                ║
║  ผลลัพธ์: ไม่มีอะไรเสร็จเลยใน 2 สัปดาห์                         ║
║                                                                  ║
║  Context Switching Cost:                                         ║
║  ทุกครั้งที่สลับงาน เสียเวลา ~15-20 นาที ในการ                  ║
║  "โหลด Context" ของงานใหม่ขึ้นมาในหัว                           ║
║                                                                  ║
║  ✅ มี WIP Limit = 1 (Focus ทีละงาน):                            ║
║                                                                  ║
║  Alice: ทำ Feature A → Done ✅ → ทำ Feature B → Done ✅           ║
║  ผลลัพธ์: ส่งมอบคุณค่าได้เร็ว และสม่ำเสมอกว่า                    ║
║                                                                  ║
║  📊 ผลการศึกษา: WIP Limit ที่เหมาะสมคือ                         ║
║     จำนวนสมาชิกทีม × 1.5 ถึง 2                                  ║
║     ทีม 4 คน → WIP Limit ประมาณ 6-8 งาน                         ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

### 5.6 Kanban Metrics

```
  📊 KANBAN METRICS — วัดสุขภาพของ Flow

  1. CYCLE TIME — เวลาตั้งแต่เริ่มทำจนเสร็จ
  ──────────────────────────────────────────
  Start (In Progress)     Done
        │                   │
        └───────────────────┘
              Cycle Time

  เป้าหมาย: ลด Cycle Time ให้สั้นและสม่ำเสมอ

  ────────────────────────────────────────────────────────────

  2. LEAD TIME — เวลาตั้งแต่ลูกค้าขอจนได้รับ
  ──────────────────────────────────────────
  Requested (Backlog)     Done
        │                   │
        └───────────────────┘
              Lead Time

  Lead Time > Cycle Time เสมอ (รวมเวลารอใน Backlog ด้วย)

  ────────────────────────────────────────────────────────────

  3. THROUGHPUT — จำนวนงานที่ทำเสร็จต่อสัปดาห์
  ──────────────────────────────────────────
  Week 1: 5 items done
  Week 2: 4 items done
  Week 3: 6 items done
  Average Throughput: 5 items/week

  ────────────────────────────────────────────────────────────

  4. CUMULATIVE FLOW DIAGRAM (CFD)
  ──────────────────────────────────────────

  Items
  ▲
  │    ████████████████████████████  DONE
  │    ██████████████████████  QA
  │    ████████████████  IN PROGRESS
  │    ████████████  TODO
  │    ██████  BACKLOG
  └───────────────────────────────────────► Time

  ถ้าแต่ละแถบกว้างขึ้น → Bottleneck ที่ Column นั้น
```

## 6. Scrum vs Kanban — เปรียบเทียบเชิงลึก

```
╔══════════════════════════════════════════════════════════════════╗
║               ⚖️  SCRUM vs KANBAN — เปรียบเทียบ                  ║
╠══════════════════════╦═══════════════════════════════════════════╣
║  มิติ                ║  SCRUM             ║  KANBAN             ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  Cadence             ║  Sprint ตายตัว     ║  Continuous Flow    ║
║                      ║  (1-4 สัปดาห์)     ║  ไม่มีรอบตายตัว     ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  Roles               ║  PO, SM, Dev       ║  ไม่กำหนด Roles     ║
║                      ║  (Required)        ║  (Optional)         ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  Commitment          ║  Sprint Commitment ║  ยืดหยุ่น           ║
║                      ║  (Sprint Backlog)  ║  เพิ่ม/ลดงานได้ตลอด ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  Iteration End       ║  Mandatory Review, ║  ไม่มีบังคับ        ║
║                      ║  Retro, Demo       ║  (ทำเมื่อต้องการ)   ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  WIP Limit           ║  ทางอ้อม (Sprint   ║  ชัดเจนต่อ Column   ║
║                      ║  Capacity)         ║  (Core Practice)    ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  Best For            ║  Product Dev,      ║  Support, Ops,      ║
║                      ║  New Features      ║  Maintenance        ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  Velocity Metric     ║  Story Points/     ║  Cycle Time,        ║
║                      ║  Sprint            ║  Throughput         ║
╠══════════════════════╬════════════════════╬═════════════════════╣
║  Change Resistance   ║  ไม่ยืดหยุ่นใน     ║  ยืดหยุ่นสูง        ║
║                      ║  Sprint (ดีมาก!)  ║  เปลี่ยนได้ตลอด     ║
╚══════════════════════╩════════════════════╩═════════════════════╝
```

### 6.1 เลือกใช้อะไรดี?

```
  🤔 DECISION GUIDE: Scrum หรือ Kanban?

  งานที่ทำมีลักษณะอย่างไร?
         │
         ▼
  มี Clear Goal ใน 1-4 สัปดาห์ ได้ไหม?
  ├── ใช่ → Scrum เหมาะกว่า
  └── ไม่ → งานเข้ามาแบบ Unpredictable ไหม?
             ├── ใช่ → Kanban เหมาะกว่า
             └── ไม่ → ลองดูทั้งสองก่อน
                       หรือใช้ Scrumban

  ────────────────────────────────────────────────────────────

  ตัวอย่างการใช้งาน:

  ✅ ใช้ SCRUM เมื่อ:
  • พัฒนา Feature ใหม่ใน Product
  • ทีมต้องการโครงสร้างและ Rhythm ที่ชัดเจน
  • ต้องการส่งมอบตาม Schedule
  • ต้องการ Regular Feedback จาก Stakeholders

  ✅ ใช้ KANBAN เมื่อ:
  • Bug Fixing / Support Requests
  • DevOps / Infrastructure Team
  • Content Creation Pipeline
  • งานที่ Interrupt ได้บ่อยและ Priority เปลี่ยนเร็ว
  • ทีมที่เพิ่งเริ่ม Agile (เริ่มง่ายกว่า Scrum)
```

## 9. Agile Metrics & Reporting

> 💡 "You can't improve what you can't measure" — Agile Metrics ช่วยให้ทีมเห็นความก้าวหน้า ระบุปัญหา และตัดสินใจได้จากข้อมูล ไม่ใช่จากความรู้สึก

### 9.1 Burndown Chart — Scrum's Most Important Chart

```
  📉 SPRINT BURNDOWN CHART

  Remaining
  Story Points
  ▲
  26│ ●                           ← Ideal Line
    │  ╲                            (เส้นทางเดิน)
  20│   ╲   ●
    │    ╲   ╲  ●
  15│     ╲      ╲ ●             ← Actual Progress
    │      ╲       ╲   ●           (เส้นความจริง)
  10│       ╲         ╲  ●
    │        ╲           ╲
   5│         ╲            ╲ ●
    │          ╲              ●
   0│           ●  ←  DONE!
    └────────────────────────────────────────────►
    Day 1   3   5   7   9   Day 10

  ──────────────────────────────────────────────────────────

  📊 วิธีอ่าน Burndown Chart:

  ✅ Actual อยู่บน/ใกล้ Ideal Line
     → ทีมทำงานตาม Plan ได้ดี

  ⚠️ Actual อยู่สูงกว่า Ideal Line มาก
     → ทีมทำงานช้ากว่าแผน มีความเสี่ยงไม่เสร็จ Sprint

  ⚠️ Actual เป็นแนวราบนาน ๆ (ไม่ลง)
     → มี Blocker ทีมทำงานไม่ได้

  ⚠️ เส้น Actual พุ่งขึ้น
     → มีงานถูกเพิ่มเข้า Sprint กลางคัน (Bad practice!)
```

```
  📊 VELOCITY CHART

  Story Points
  Completed
  ▲
  35│          █                    ← High but Outlier
  30│  █       █   █
    │  █   █   █   █   █
  25│  █   █   █   █   █   █
    │  █   █   █   █   █   █

      SP1  SP2  SP3  SP4  SP5  SP6

  ────────────────────────────────────────────────────────────
  การใช้ Velocity:
  • ทำนายว่า Sprint ต่อไปจะทำได้กี่ Points
  • วางแผน Release Date โดยคำนวณจาก Backlog ÷ Velocity
  • ตรวจจับ Trend (Velocity เพิ่ม = ทีมแข็งแกร่งขึ้น)

  ⚠️ ข้อควรระวัง:
  • Velocity เป็นเครื่องมือวางแผน ไม่ใช่วัด Performance ทีม

  • Story Points ของแต่ละทีมไม่เท่ากัน
```

### 9.3 Cumulative Flow Diagram — Kanban's Power Metric

```
  📈 CUMULATIVE FLOW DIAGRAM (CFD)

  Tasks
  ▲
  50│████████████████████████████████ DONE
    │██████████████████████████  QA
  40│████████████████████  REVIEW
    │████████████████  IN PROGRESS
  30│████████████  TODO
    │████████  BACKLOG
  20│████  ANALYSIS
    │
  10│
    │
   0└─────────────────────────────────────────────────────►
     Week1   Week2   Week3   Week4   Week5   Week6

  ────────────────────────────────────────────────────────────
  วิธีอ่าน CFD:

  ✅ แต่ละแถบสีมีความกว้างสม่ำเสมอ → Flow ดี ไม่มี Bottleneck

  ❌ แถบ IN PROGRESS กว้างขึ้นเรื่อย ๆ → Bottleneck ที่ Code Review
     → WIP สูงเกินไป งานค้างอยู่ใน Stage นั้นนาน

  ❌ แถบ DONE ไม่ขึ้น (แนวนอน) → งานไม่ผ่านสู่ Done
     → ปัญหาที่ Process ท้ายสุด (QA หรือ Deploy)
```

## 10. ปัญหาที่พบบ่อยใน Agile

> 💡 Agile ที่ทำแบบผิด ๆ มีชื่อเรียกว่า "Dark Agile" หรือ "Zombie Scrum" — รู้จักปัญหาเหล่านี้เพื่อหลีกเลี่ยง

### 10.1 Anti-patterns ที่พบบ่อย

```
╔══════════════════════════════════════════════════════════════════╗
║            ⚠️  COMMON AGILE ANTI-PATTERNS                        ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  🧟 ZOMBIE SCRUM                                                 ║
║  → ทำตาม Ceremony ครบ แต่ไม่ Inspect หรือ Adapt อะไรเลย        ║
║  → Sprint Retro ทุกครั้ง แต่ไม่มี Action Item ที่ทำจริง          ║
║  ─────────────────────────────────────────────────              ║
║  แก้: Retro ต้องมี 1-3 Action Items ที่ทำได้ใน Sprint ต่อไป     ║
║                                                                  ║
║  💼 SCRUM BUT (Scrumbut)                                         ║
║  → "เราทำ Scrum แต่ว่า... ไม่มี Sprint Review"                   ║
║  → "เราทำ Scrum แต่ว่า... PO ไม่ค่อย Available"                  ║
║  → "เราทำ Scrum แต่ว่า... Sprint ของเราคือ 2 เดือน"              ║
║  ─────────────────────────────────────────────────              ║
║  แก้: ถ้า Scrum ไม่เหมาะ ลองใช้ Kanban หรือ Hybrid แทน         ║
║                                                                  ║
║  🔩 WATERFALL IN SPRINT                                          ║
║  → Sprint ทำตามลำดับ: Analyze → Design → Code → Test ครบ        ║
║    โดยแต่ละ Task ทำได้ในช่วงเวลาเท่านั้น (เหมือน Mini-Waterfall)║
║  ─────────────────────────────────────────────────              ║
║  แก้: ทำงานพร้อมกัน Cross-functional ไม่ใช่ Sequential          ║
║                                                                  ║
║  📝 STORY POINTS THEATER                                         ║
║  → ทีม Estimate เพื่อให้ Management พอใจ ไม่ใช่เพื่อวางแผนจริง  ║
║  → PO กดดันให้ Estimate ต่ำเพื่อให้ Sprint ดูน่าประทับใจ         ║
║  ─────────────────────────────────────────────────              ║
║  แก้: Story Points คือเครื่องมือทีม ไม่ใช่ตัวชี้วัด Management  ║
║                                                                  ║
║  🏃 SPRINT 0 ตลอดไป                                              ║
║  → ทีมไม่ยอมเริ่ม Sprint จริง ๆ เพราะรู้สึกว่ายัง "ไม่พร้อม"    ║
║  ─────────────────────────────────────────────────              ║
║  แก้: Ship Something Small เร็วที่สุด แล้วค่อยปรับปรุง            ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

### 10.2 Tips สำหรับทีมที่เพิ่งเริ่ม Agile

```
  💡 PRACTICAL TIPS สำหรับทีมใหม่

  เดือนที่ 1: เริ่มง่าย ๆ
  ────────────────────────
  ✅ เริ่มด้วย Visual Board (Trello/Jira Kanban) ง่าย ๆ
  ✅ ทำ Daily Standup ทุกวัน แม้แค่ 10 นาที
  ✅ กำหนด WIP Limit ที่ทุกคน agree (เริ่มที่ n+1 ก่อน)
  ✅ ทำ Weekly Retro แบบง่าย: เพิ่ม 1 สิ่ง, ลด 1 สิ่ง

  เดือนที่ 2-3: เพิ่ม Structure
  ──────────────────────────────
  ✅ ลอง Sprint 2 สัปดาห์ ถ้าเหมาะกับงาน
  ✅ ทำ Sprint Planning อย่างจริงจัง
  ✅ เริ่ม Track Velocity
  ✅ ทำ Demo ให้ Stakeholders ทุก Sprint

  เดือนที่ 4+: ปรับปรุงต่อเนื่อง
  ─────────────────────────────────
  ✅ วิเคราะห์ Metrics: Burndown, Velocity, Cycle Time
  ✅ ทดลอง Engineering Practices: TDD, Pair Programming
  ✅ Integrate CI/CD เข้ากับ Agile Workflow
  ✅ ปรับ Process จาก Retro อย่างต่อเนื่อง
```
