## 1. Agile คืออะไร? <a name="1"></a>

> 💡 **Agile** ไม่ใช่เครื่องมือ ไม่ใช่ขั้นตอน แต่คือ **แนวคิด (Mindset)** ในการพัฒนาซอฟต์แวร์ที่เน้นความยืดหยุ่น การส่งมอบคุณค่าอย่างต่อเนื่อง และการทำงานร่วมกันของทีม

### 1.1 ปัญหาของวิธีเดิม — ทำไมถึงต้องมี Agile?

```
  ❌ กรณีศึกษา: โปรเจกต์ระบบ ERP ด้วยวิธี Waterfall แบบดั้งเดิม

  มกราคม        มีนาคม         มิถุนายน        ตุลาคม         ธันวาคม
  ────────────  ─────────────  ──────────────  ─────────────  ───────────
  วิเคราะห์      ออกแบบ          พัฒนา           ทดสอบ          ส่งมอบ
  Requirements  Architecture   (4 เดือน)        UAT            ❌ FAIL
  (2 เดือน)    (3 เดือน)
       │
       └─► ลูกค้าบอก "ต้องการระบบ Payroll"
                │
                ▼
       ตุลาคม: ลูกค้าเปลี่ยนใจ → "เพิ่ม Module HR ด้วย"
                │
                ▼
       ธันวาคม: ส่งมอบระบบที่ลูกค้าไม่ต้องการอีกแล้ว
               ใช้เงิน 12 ล้านบาท ใช้เวลา 12 เดือน
               แต่ Business เปลี่ยนไปแล้วทั้งหมด

  ─────────────────────────────────────────────────────────────────
  สรุปปัญหาหลักของ Waterfall:

  ❌ รับ Feedback ช้า   → รู้ว่าพลาดตอน Deploy แล้วเท่านั้น
  ❌ เปลี่ยนแปลงยาก    → Cost สูงมากถ้าเปลี่ยน Requirement ระหว่างทาง
  ❌ ส่งมอบช้า          → ลูกค้าเห็นผลลัพธ์จริงครั้งแรกในเดือนที่ 12
  ❌ ความเสี่ยงสูง      → ลงทุนเต็มที่ก่อน แล้วค่อยรู้ว่าถูกผิดทีหลัง
```

## 4. Scrum Framework <a name="4"></a>

> 💡 **Scrum** คือ Lightweight Framework ที่ช่วยให้คน ทีม และองค์กรสร้างคุณค่าผ่านการ Adapt กับปัญหาที่ซับซ้อน มีโครงสร้างชัดเจน: **3 Roles + 5 Events + 3 Artifacts**

### 4.1 Big Picture ของ Scrum

```

  ┌──────────────────────────────────────────────────────────────┐

  │  │  (งานใน Sprint)   │    (15m)    │      (Coding,        │  │

#### Event 4: Sprint Review

```
  🎭 SPRINT REVIEW — Demo งานให้ Stakeholders

  ┌────────────────────────────────────────────────────────────────┐

  ───────────────────
  Step 1: PO แจ้ง Items ที่ "Done" ใน Sprint นี้
     │
  Step 2: Developers Demo Working Software (ไม่ใช่ Slides!)
     │
  Step 3: Stakeholders ให้ Feedback โดยตรง
     │
  Step 4: PO ปรับ Product Backlog ตาม Feedback
     │
  Step 5: วางแผน Sprint ต่อไปร่วมกัน

  ─────────────────────────────────────────────────────────────────

  ❌ Sprint Review ≠ Approval Meeting
     ไม่ใช่ขอ "Pass/Fail" จาก Management

  ✅ Sprint Review = Collaboration Session
     เพื่อให้ Product ดีขึ้นร่วมกัน

  │   I want [some goal],                                           │
  │   so that [some reason/benefit]"                                │
  └─────────────────────────────────────────────────────────────────┘

  ตัวอย่าง:
  ──────────
  As a registered user,
  I want to reset my password via email,
  so that I can regain access when I forget my password.

  Acceptance Criteria:
  ☐ เมื่อกด "Forgot Password" จะมีหน้าให้กรอก Email
  ☐ ระบบส่ง Email พร้อม Reset Link ภายใน 1 นาที
  ☐ Reset Link มีอายุ 24 ชั่วโมงเท่านั้น
  ☐ หลังรีเซ็ตสำเร็จ Link เก่าใช้งานไม่ได้อีก
  ☐ แสดง Error ที่เหมาะสมเมื่อ Email ไม่มีในระบบ

  ─────────────────────────────────────────────────────────────────
  💡 INVEST Criteria — User Story ที่ดีต้องเป็น:

  ┌─────┬────────────────┬────────────────────────────────────────┐
  │  I  │ Independent    │ แต่ละ Story เป็นอิสระต่อกัน            │
  │  N  │ Negotiable     │ ปรับเปลี่ยนได้ตาม Conversation         │
  │  V  │ Valuable       │ สร้างคุณค่าให้ผู้ใช้หรือ Business      │
  │  E  │ Estimable      │ ประมาณขนาดงานได้                       │
  │  S  │ Small          │ เล็กพอทำเสร็จใน 1 Sprint               │
  │  T  │ Testable       │ ทดสอบได้ว่าทำเสร็จแล้วจริง              │
  └─────┴────────────────┴────────────────────────────────────────┘
```

### 4.9 Story Points & Planning Poker

```
  🃏 PLANNING POKER — วิธี Estimate ที่นิยมใน Scrum

  ทำไมต้อง Story Points ไม่ใช่ ชั่วโมง?
  ───────────────────────────────────────
  ❌ ชั่วโมง: "Login Page ทำ 4 ชั่วโมง"
     → แต่ละคนทำได้ไม่เท่ากัน ขึ้นอยู่กับประสบการณ์

  ✅ Story Points: "Login Page = 3 Points" (ความซับซ้อนสัมพัทธ์)
     → ถ้า Register = 5 Points แสดงว่า Register ยากกว่า Login ~67%
     → ทีมเรียนรู้ Velocity ของตัวเองตามเวลา

  ─────────────────────────────────────────────────────────────────
  วิธีเล่น Planning Poker:
  ─────────────────────────
  ┌──────────────────────────────────────────────────────────────┐
  │  1. PO อ่าน User Story ให้ทีมฟัง                              │
  │  2. ทีมถามคำถามเพื่อทำความเข้าใจ                              │
  │  3. ทุกคนเลือก Card พร้อมกัน (เลข Fibonacci)                  │
  │  4. เปิด Card พร้อมกัน — ห้ามบอกก่อน!                         │
  │  5. ถ้าค่าใกล้กัน → ใช้ค่ากลาง                                │
  │  6. ถ้าค่าต่างกันมาก → คนที่สูงสุดและต่ำสุดอธิบายเหตุผล       │
  │  7. Vote ใหม่จนได้ Consensus                                  │
  └──────────────────────────────────────────────────────────────┘

  Fibonacci Cards:
  ┌───┬───┬───┬───┬───┬────┬────┬────┬────┐
  │ 1 │ 2 │ 3 │ 5 │ 8 │ 13 │ 21 │ 34 │  ? │
  └───┴───┴───┴───┴───┴────┴────┴────┴────┘
  ? = ต้องการข้อมูลเพิ่มก่อน Estimate

  ─────────────────────────────────────────────────────────────────
  ตัวอย่างการ Estimate:

  Story: "ผู้ใช้สามารถ Login ด้วย Social Media (Google/Facebook)"

  Alice: [8]  "ต้องทำ OAuth Integration สองเจ้า ซับซ้อนมาก"
  Bob:   [5]  "เราน่าจะมี Library ช่วยได้"
  Charlie:[13] "ต้องทดสอบหลาย Edge Cases"

  → Alice และ Charlie อธิบายเหตุผล
  → ทีม Vote ใหม่ → ได้ [8] โดย Consensus

## 5. Kanban Method <a name="5"></a>

> 💡 **Kanban** (看板) มาจากภาษาญี่ปุ่น แปลว่า "กระดานสัญญาณ" มีต้นกำเนิดจากระบบการผลิต Toyota ในทศวรรษ 1950 และถูกนำมาปรับใช้ใน Software Development โดย David J. Anderson ในปี 2007

### 5.1 ต้นกำเนิดของ Kanban

```
  📜 KANBAN — เส้นทางสู่ Software Development

  1950s                    2007                      ปัจจุบัน
  ─────────────────────    ──────────────────────    ─────────────────
  Toyota Production        David J. Anderson         ทีม Software ทั่วโลก
  System (TPS)             นำ Kanban มาใช้ใน         ใช้ Kanban ในงาน
                           Software Development      Knowledge Work
  "ผลิตเฉพาะสิ่งที่        "Visualize งาน,           Bug Fixes, DevOps,
   ต้องการ เมื่อ            Limit WIP,                Support, Ops,
   ต้องการ"                 Flow-based"               Content, Design

  Just-in-time       →    Pull System       →    Continuous Flow
  Manufacturing            for Software           Improvement
```

### 5.2 Kanban Core Principles — 4 หลักการ

```
╔══════════════════════════════════════════════════════════════════════╗
║                  🎌 KANBAN CORE PRINCIPLES                           ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  1️⃣  START WITH WHAT YOU DO NOW                                      ║
║      เริ่มจากสิ่งที่ทำอยู่ปัจจุบัน                                    ║
║      → ไม่ต้องเปลี่ยนทุกอย่างทันที                                   ║
║      → Visualize งานที่ทำอยู่แล้วก่อน                                ║
║      → Kanban ปรับตัวเข้ากับ Process ที่มีอยู่                       ║
║                                                                      ║
║  2️⃣  AGREE TO PURSUE INCREMENTAL, EVOLUTIONARY CHANGE               ║
║      เปลี่ยนแปลงทีละเล็กทีละน้อย ไม่ใช่ Big Bang                     ║
║      → ทดลอง → วัดผล → ปรับปรุง → ทดลองต่อ                          ║
║                                                                      ║
║  3️⃣  RESPECT CURRENT PROCESS, ROLES, RESPONSIBILITIES               ║
║      เคารพสิ่งที่ทำงานได้ดีอยู่แล้ว                                   ║
║      → ไม่บังคับเปลี่ยน Job Title หรือ Role ทันที                    ║
║      → ค่อย ๆ ปรับตามความเข้าใจที่เพิ่มขึ้น                          ║
║                                                                      ║
║  4️⃣  ENCOURAGE ACTS OF LEADERSHIP AT ALL LEVELS                     ║
║      ทุกคนในทีมมีส่วนร่วมในการปรับปรุง                                ║
║      → ไม่ใช่แค่ผู้บริหารที่ตัดสินใจ                                  ║
║      → Developer ก็ Lead การเปลี่ยนแปลงได้                           ║
║                                                                      ║
╚══════════════════════════════════════════════════════════════════════╝
```

### 5.5 WIP Limit — กุญแจสำคัญของ Kanban

```
╔══════════════════════════════════════════════════════════════════════╗
║       🚦 WIP LIMIT (Work In Progress Limit) — ทำไมถึงสำคัญ?         ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  📐 Little's Law (กฎของ Little):                                    ║
║                                                                      ║
║         Cycle Time = WIP ÷ Throughput                               ║
║                                                                      ║
║  → ลดงานที่ทำพร้อมกัน = Cycle Time สั้นลง = ส่งมอบเร็วขึ้น          ║
║                                                                      ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  ❌ ไม่มี WIP Limit — Context Switching Problem:                     ║
║                                                                      ║
║  Alice: กำลังทำ Feature A (30%), B (20%), C (10%)                   ║
║                                                                      ║
║  Week 1: A 30% → สลับ → B 20% → สลับ → C 10% → สลับ → A 50%       ║
║  Week 2: ไม่มีอะไรเสร็จเลย!                                         ║
║                                                                      ║
║  Context Switching Cost:                                             ║
║  ทุกครั้งที่สลับงาน เสียเวลา 15-20 นาที ในการ "โหลด Context"        ║
║  ของงานใหม่ขึ้นมาในหัว                                               ║
║                                                                      ║
║  ✅ มี WIP Limit = 1 — Focus ทีละงาน:                                ║
║                                                                      ║
║  Week 1: Feature A ────────────────────────────────────► Done ✅    ║
║  Week 2: Feature B ──────────────────────────► Done ✅               ║
║                                                                      ║
║  → ส่งมอบคุณค่าได้เร็วและสม่ำเสมอกว่า                               ║
║                                                                      ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  💡 WIP Limit ที่แนะนำ: จำนวนสมาชิก × 1.5 ถึง 2                   ║
║     ทีม 4 คน → WIP Limit ประมาณ 6-8 งาน                            ║
║                                                                      ║
║  🎯 หลักการ: "Stop Starting, Start Finishing"                        ║
║     หยุดเริ่มงานใหม่ → โฟกัสทำงานเดิมให้เสร็จ                        ║
║                                                                      ║
╚══════════════════════════════════════════════════════════════════════╝
```

### 5.7 Kanban Metrics

```
  📊 KANBAN METRICS — 4 ตัวชี้วัดสำคัญ

  ┌─────────────────────────────────────────────────────────────────┐
  │  1. CYCLE TIME — เวลาตั้งแต่เริ่มทำจนเสร็จ                      │
  │                                                                 │
  │  Start (In Progress)              Done                          │
  │         │                           │                          │
  │         └───────────────────────────┘                          │
  │                    Cycle Time                                   │
  │                                                                 │
  │  เป้าหมาย: Cycle Time สั้นและ สม่ำเสมอ                          │
  ├─────────────────────────────────────────────────────────────────┤
  │  2. LEAD TIME — เวลาตั้งแต่ลูกค้าขอจนได้รับ                      │
  │                                                                 │
  │  Requested (Backlog)              Done                          │
  │         │                           │                          │
  │         └───────────────────────────┘                          │
  │                     Lead Time                                   │
  │                                                                 │
  │  Lead Time > Cycle Time เสมอ (รวมเวลารอใน Backlog)              │
  ├─────────────────────────────────────────────────────────────────┤
  │  3. THROUGHPUT — จำนวนงานที่ทำเสร็จต่อสัปดาห์                    │
  │                                                                 │
  │  Week 1: ████████████ 6 items                                   │
  │  Week 2: ██████████ 5 items                                     │
  │  Week 3: ████████████ 6 items                                   │
  │  Week 4: ████████ 4 items                                       │
  │  Average Throughput: 5.25 items/week                            │
  ├─────────────────────────────────────────────────────────────────┤
  │  4. FLOW EFFICIENCY — สัดส่วนเวลาที่งาน "ไหล" จริง ๆ            │
  │                                                                 │
  │  Flow Efficiency = Active Time ÷ Lead Time × 100               │
  │                                                                 │
  │  ถ้า Feature ใช้เวลา 10 วัน แต่ทำงานจริงแค่ 3 วัน              │
  │  Flow Efficiency = 3/10 × 100 = 30%                             │
  │  → 70% เป็นเวลารอ (Waiting, Blocked, Queue)                    │
  └─────────────────────────────────────────────────────────────────┘
```

## 10. ปัญหาที่พบบ่อยใน Agile <a name="10"></a>

> 💡 Agile ที่ทำแบบผิด ๆ มีชื่อเรียกว่า **"Dark Agile"** หรือ **"Zombie Scrum"** — รู้จักเพื่อหลีกเลี่ยง!

### 10.1 Anti-patterns ที่พบบ่อย

```
╔══════════════════════════════════════════════════════════════════════╗
║                  ⚠️  COMMON AGILE ANTI-PATTERNS                      ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  🧟 ZOMBIE SCRUM                                                     ║
║  ────────────────────────────────────────────────────────────────  ║
║  อาการ: ทำตาม Ceremony ครบ แต่ไม่ Inspect หรือ Adapt อะไรเลย       ║
║          Sprint Retro ทุกครั้ง แต่ไม่มี Action Item จริง            ║
║                                                                      ║
║  ตัวอย่าง: "เราทำ Retro ทุก Sprint มา 6 เดือนแล้ว แต่ปัญหาเดิม     ║
║             ยังอยู่ทุกอัน"                                           ║
║                                                                      ║
║  แก้: Retro ต้องมี 1-3 Action Items ที่ทำได้จริงใน Sprint ถัดไป    ║
║                                                                      ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  💼 SCRUM BUT (Scrumbut)                                             ║
║  ────────────────────────────────────────────────────────────────  ║
║  อาการ: "เราทำ Scrum แต่ว่า..."                                      ║
║                                                                      ║
║  ❌ "แต่ว่า ไม่มี Sprint Review เพราะไม่มีเวลา"                      ║
║  ❌ "แต่ว่า PO ไม่ค่อย Available ใน Sprint Planning"                 ║
║  ❌ "แต่ว่า Sprint ของเราคือ 2 เดือน ไม่ใช่ 2 สัปดาห์"               ║
║  ❌ "แต่ว่า Manager สั่ง Dev โดยตรงโดยไม่ผ่าน Backlog"              ║
║                                                                      ║
║  แก้: ถ้า Scrum ไม่เหมาะ ลองใช้ Kanban หรือ Hybrid แทน             ║
║       หยุดแกล้งทำ ทำของจริงหรือเปลี่ยนวิธี                           ║
║                                                                      ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                      ║
║  🔩 WATERFALL IN SPRINT (Mini-Waterfall)                             ║
║  ────────────────────────────────────────────────────────────────  ║
║  อาการ: Sprint ทำตามลำดับ Analysis→Design→Code→Test                 ║
║          แต่ละ Phase ทำโดยคนละกลุ่ม ไม่ทำพร้อมกัน                   ║
║                                                                      ║
║  ❌ Week 1: BA เขียน Requirements                                    ║
║  ❌ Week 2: Designer ออกแบบ                                          ║
║  ❌ Week 3: Dev Code                                                 ║
║  ❌ Week 4: QA Test → ไม่มีเวลาแก้ Bug!                              ║
║                                                                      ║
║  แก้: ทำงาน Cross-functional ไม่ใช่ Sequential                       ║
║       ทุกคนช่วยกันทำงานให้เสร็จ ไม่รอทีละ Phase                      ║
║                                                                      ║

### 10.2 Tips สำหรับทีมที่เพิ่งเริ่ม Agile

```
  💡 AGILE ADOPTION ROADMAP — 3 เดือนแรก

  เดือนที่ 1: VISUALIZE (ทำให้มองเห็น)
  ───────────────────────────────────────
  ✅ สร้าง Visual Board (Trello/Jira Kanban) ง่าย ๆ
  ✅ ทำ Daily Standup ทุกวัน แม้แค่ 10 นาที
  ✅ กำหนด WIP Limit ที่ทุกคน Agree
  ✅ ทำ Weekly Retro แบบง่าย: เพิ่ม 1 สิ่ง, ลด 1 สิ่ง

  เดือนที่ 2: STRUCTURE (เพิ่มโครงสร้าง)
  ──────────────────────────────────────────
  ✅ ลอง Sprint 2 สัปดาห์ ถ้าเหมาะกับงาน
  ✅ ทำ Sprint Planning อย่างจริงจัง
  ✅ เริ่ม Track Velocity
  ✅ ทำ Demo ให้ Stakeholders ทุก Sprint

  เดือนที่ 3: OPTIMIZE (ปรับปรุงต่อเนื่อง)
  ─────────────────────────────────────────
  ✅ วิเคราะห์ Metrics: Burndown, Velocity, Cycle Time
  ✅ ทดลอง Engineering Practices เช่น TDD, Pair Programming
  ✅ Integrate CI/CD เข้ากับ Agile Workflow
  ✅ ปรับ Process จาก Retro อย่างต่อเนื่อง

  ─────────────────────────────────────────────────────────────────

  🎯 Golden Rules สำหรับทีมใหม่:
  1. เริ่มง่าย ๆ แล้วค่อยเพิ่ม Complexity
  2. ทำอย่างสม่ำเสมอดีกว่าทำสมบูรณ์แบบแต่ไม่สม่ำเสมอ
  3. Retrospective คือโอกาสทอง อย่าข้ามเด็ดขาด
  4. ฉลอง Small Wins เพื่อรักษา Momentum
```

### 11.1 Case Study: Startup E-Commerce — ใช้ Scrum

```
  🛒 CASE STUDY: ShopTH Startup — 6 สมาชิก

  บริบท:
  ─────────────────────────────────────────────────────────────────
  • Startup E-Commerce ขนาดเล็ก 6 คน
  • Product: แอพสั่งซื้อสินค้าออนไลน์
  • ปัญหาเดิม: ไม่มี Structure, งานค้าง, ลูกค้าไม่รู้ Progress

  การนำ Scrum มาใช้:
  ─────────────────────────────────────────────────────────────────
  Sprint Duration: 2 สัปดาห์
  Team: 1 PO (CEO), 1 SM (Tech Lead), 4 Dev (2 FE, 1 BE, 1 QA)

  ผลลัพธ์หลัง 3 เดือน:
  ─────────────────────────────────────────────────────────────────
  Before Scrum:               After Scrum (6 Sprints):
  ─────────────────────       ──────────────────────────────────
  Feature ส่งมอบ: 2/เดือน    Feature ส่งมอบ: 8/เดือน  (+300%)
  Bug ใน Production: 15/เดือน Bug ใน Production: 4/เดือน (-73%)
  Estimate Accuracy: 40%      Estimate Accuracy: 78%  (+95%)
  Team Happiness: 3/5         Team Happiness: 4.2/5   (+40%)

  Lessons Learned:
  ─────────────────────────────────────────────────────────────────
  ✅ Sprint Review ช่วยให้ลูกค้าเห็น Progress และให้ Feedback เร็ว
  ✅ Daily Scrum ช่วยระบุ Blocker ก่อนที่จะกลายเป็นวิกฤต
  ✅ Retrospective Sprint ที่ 3 เปิดเผยว่า QA ทำงาน Overload
     → ปรับ WIP Limit และเพิ่ม Automation Testing
```

### 11.2 Case Study: Tech Company — ใช้ Kanban (DevOps Team)

```
  🔧 CASE STUDY: TechCorp DevOps Team — 4 สมาชิก

  บริบท:
  ─────────────────────────────────────────────────────────────────
  • ทีม DevOps รับงานหลายประเภทพร้อมกัน
  • Infrastructure Requests จาก Dev Team
  • Emergency Production Issues
  • Planned Maintenance
  • Security Patches

  ปัญหาเดิม (ก่อนใช้ Kanban):
  ─────────────────────────────────────────────────────────────────
  • ทีมรับงานทุกอย่างพร้อมกัน → ไม่มีอะไรเสร็จ
  • Priority ไม่ชัด → งาน Emergency รอนานเกินไป
  • ไม่รู้ว่า Request ไหนอยู่ที่ขั้นตอนไหน

  Kanban Board ที่ออกแบบ:
  ─────────────────────────────────────────────────────────────────
  ┌──────────┬──────────┬────────────┬──────────┬────────────┐
  │ INCOMING │ SELECTED │IN PROGRESS │  VERIFY  │    DONE    │
  │          │ (WIP: 6) │  (WIP: 4)  │ (WIP: 2) │            │
  │ Priority │          │            │          │            │
  │ 🔴 P1    │          │            │          │            │
  │ 🟠 P2    │          │            │          │            │
  │ 🟡 P3    │          │            │          │            │
  └──────────┴──────────┴────────────┴──────────┴────────────┘

  Policy สำหรับ Emergency (P1):
  → หยุด P2/P3 ที่กำลังทำ
  → Emergency เข้า IN PROGRESS ทันที
  → แจ้ง Stakeholders ทันที

  ผลลัพธ์หลัง 2 เดือน:
  ─────────────────────────────────────────────────────────────────
  Avg Cycle Time P1: 4 ชม. → 1.5 ชม. (-62%)
  Avg Cycle Time P3: 5 วัน → 3 วัน (-40%)
  Throughput: 8 items/week → 13 items/week (+63%)
```

     → สัญญาณ BAD Practice
     → งานใหม่ควรใส่ใน Product Backlog รอ Sprint ถัดไป
     ยกเว้นเป็น Critical Bug ที่กระทบ Sprint Goal

  3. จาก Trend: ทีมน่าจะทำเสร็จได้ (เส้นกำลังลงเร็ว)
     แต่เส้นอยู่บน Ideal Line อยู่เล็กน้อย
     SM ควร: Monitor วันต่อวัน หา Risk ที่ยังอาจเกิด
     และแจ้ง PO ถ้า Sprint Goal มีความเสี่ยง
```

### ✏️ Exercise 5: Mini-Project — สร้าง Kanban Board ด้วย Trello

```
  🎯 Hands-on Exercise: ใช้ Trello สร้าง Kanban Board

  ขั้นตอน:
  ─────────────────────────────────────────────────────────────────
  Step 1: สมัคร Trello (ฟรี) ที่ trello.com

  Step 2: สร้าง Board ใหม่ชื่อ "My Agile Board"

  Step 3: สร้าง Lists (Columns):
          ☐ 📋 Backlog
          ☐ 📌 To Do (Sprint)
          ☐ 🔨 In Progress (Limit: 2 cards)
          ☐ 👀 In Review (Limit: 1 card)
          ☐ ✅ Done

  Step 4: เพิ่ม User Stories เป็น Cards ใน Backlog
          (ใช้ Stories จาก Exercise 2 หรือสร้างเอง)

  Step 5: กำหนด WIP Limit โดย:
          ใส่ WIP Limit ใน List Name เช่น "In Progress (Limit: 2)"
          หรือใช้ Power-Up "WIP Limit" ถ้ามี

  Step 6: จำลองการทำงาน 1 Sprint (2 สัปดาห์) โดย:
          ☐ เลือก Cards จาก Backlog → To Do
          ☐ Move Cards ตามสถานะงาน
          ☐ เพิ่ม Label สีตาม Priority
          ☐ เพิ่ม Due Date และ Assignee

  Step 7: Screenshot Board และ Submit พร้อม Comment:
          - WIP Limit ที่กำหนดและเหตุผล
          - ปัญหาที่พบระหว่างทำ
          - สิ่งที่เรียนรู้
```
