## 1. Product Development Tracking คืออะไร? <a name="1"></a>

> 💡 **เปรียบง่าย ๆ :** ถ้าทีม Dev คือนักเดินทาง และ Software คือปลายทาง  
> **Tracking Tool** คือ GPS — บอกว่าตอนนี้อยู่ที่ไหน เหลืออีกนานแค่ไหน และมีอุปสรรคข้างหน้าไหม

> 💡 **Product Development Tracking** คือกระบวนการติดตามความคืบหน้าของการพัฒนา Software ตั้งแต่ Idea จนถึง Deployment โดยใช้ Tools เพื่อให้ทุกคนในทีม รวมถึง Stakeholders เห็นภาพรวมเดียวกันตลอดเวลา

### 1.1 มองภาพรวมก่อน: Software Project ไม่มี Tracking เป็นอย่างไร?

### 1.3 วงจรชีวิตของ Work Item

```
  🔄 WORK ITEM LIFECYCLE — จาก Idea สู่ Done

  ──────────────────────────────────────────────────────────────────
   💡 IDEA        📝 BACKLOG       🏃 SPRINT        🚀 DEPLOYED
  ──────────────────────────────────────────────────────────────────
   ลูกค้าขอ   →  PO เขียนเป็น →  Dev รับและ  →   ผู้ใช้ได้ใช้
   Bug พบ        User Story        พัฒนา          จริง ๆ
   ทีมเสนอ       Estimate          Test ผ่าน
                 Priority                      
  ──────────────────────────────────────────────────────────────────
              TRACKING ครอบคลุมตลอดทุกช่วง
  ──────────────────────────────────────────────────────────────────

  คำถามที่ Tracking Tool ต้องตอบได้ตลอดเวลา:

  ┌─────────────────────────────────────────────────────────────────┐
  │  ❓ งานนี้อยู่ที่ขั้นตอนไหน?    → ดูที่ Status ของ Issue        │
  │  ❓ ใครรับผิดชอบอยู่?           → ดูที่ Assignee                │
  │  ❓ จะเสร็จเมื่อไหร่?           → ดู Due Date / Sprint          │
  │  ❓ มีอุปสรรคอะไรไหม?           → ดู Blocked Flag / Comment     │
  │  ❓ Release นี้คืบหน้าแค่ไหน?  → ดู Burndown / Progress Chart   │
  └─────────────────────────────────────────────────────────────────┘
```

## 2. ทำไมต้อง Track งาน? <a name="2"></a>

### 2.1 ต้นทุนของการไม่ Track (เรื่องจริงที่เกิดทุกวัน)

```
  ❌ สถานการณ์จริง: ทีม 5 คน ไม่มี Tracking Tool

  ─────────────────────────────────────────────────────────────────
  วันจันทร์ 09:00 (Stand-up Meeting):
  PM:      "งาน Login Feature เสร็จหรือยัง?"
  Alice:   "ฉันคิดว่า Bob กำลังทำอยู่นะ"
  Bob:     "เปล่า ฉันคิดว่า Charlie รับไป"
  Charlie: "ไม่ได้รับเลย ฉันทำ Payment อยู่"
  → ❌ ไม่มีใครรู้สถานะ ใช้เวลาหา 30 นาที

  วันพุธ (Client Call):
  Client: "Feature ที่ขอใน Email 2 สัปดาห์ก่อนเป็นยังไงบ้าง?"
  PM:     "...ขอเช็คก่อนนะครับ"
  Dev:    "ไม่รู้ว่า Email ไหน ส่งให้ดูหน่อย"
  → ❌ ข้อมูลกระจัดกระจาย ตามไม่ได้

  วันศุกร์ (ก่อน Release):
  "เราพร้อม Release ไหม?"
  → ❌ ไม่มีใครรู้ว่างานไหนเสร็จ งานไหนค้างอยู่
     ต้องนั่งถามทีละคน ใช้เวลา 2 ชั่วโมง

  ─────────────────────────────────────────────────────────────────
  💸 ต้นทุนของการไม่ Track (ต่อสัปดาห์/คน):

  ┌──────────────────────────────────────────┬───────────────────┐
  │ กิจกรรมที่สูญเสียเวลา                   │ ชั่วโมงที่เสียไป  │
  ├──────────────────────────────────────────┼───────────────────┤
  │ เวลาตามหาสถานะงาน                       │ ~3 ชั่วโมง        │
  │ Meeting ที่ไม่มีประสิทธิภาพ              │ ~2 ชั่วโมง        │
  │ งานที่ทำซ้ำ (ไม่รู้ว่าคนอื่นทำแล้ว)    │ ~1 ชั่วโมง        │
  ├──────────────────────────────────────────┼───────────────────┤
  │ รวมเวลาสูญเสียต่อคน                     │ ~6 ชั่วโมง/สัปดาห์│
  └──────────────────────────────────────────┴───────────────────┘

  คำนวณ: 5 คน × 6 ชม./สัปดาห์ = 30 ชม./สัปดาห์ ≈ 1 FTE สูญเปล่า!
```

```
  │  ─────────────────────────────────────────────────────         │
  │  • ทุกคนเห็นสถานะงานเดียวกัน Real-time                         │
  │  • PO ดู Progress โดยไม่ต้องถาม Dev ทุกวัน                     │
  │  • Stakeholders ติดตามได้ด้วยตัวเอง                              │
  ├─────────────────────────────────────────────────────────────────┤
  │  🎯 ACCOUNTABILITY (ความรับผิดชอบ)                               │
  │  ─────────────────────────────────────────────────────         │
  │  • ชัดเจนว่าใครรับผิดชอบงานไหน                                  │
  │  • ประวัติการเปลี่ยนแปลงถูกบันทึกอัตโนมัติ                       │
  │  • ไม่มีข้อแก้ตัว "ฉันไม่รู้ว่าต้องทำ"                           │

  │  🔮 PREDICTABILITY (การคาดการณ์)                                │

  │  • วางแผน Release Date ได้แม่นยำขึ้น                            │

  ├─────────────────────────────────────────────────────────────────┤
  │  ⚡ EFFICIENCY (ประสิทธิภาพ)                                     │
  │  ─────────────────────────────────────────────────────         │
  │  • ลด Meeting ที่ไม่จำเป็น                                       │
  │  • ค้นหาข้อมูลเก่าได้ทันที                                       │
  │  • Prioritize งานได้ถูกต้อง                                      │
  └─────────────────────────────────────────────────────────────────┘
```

### 2.3 ROI ของ Tracking Tool (Return on Investment)

```
  📈 ตัวอย่างการคำนวณ ROI ของการใช้ Jira (ทีม 8 คน)

  ┌─────────────────────────────────────────────────────────────────┐
  │  ก่อนใช้ Tracking Tool                                          │
  │  ─────────────────────────────────────────────────────         │
  │  เวลาสูญเสีย/สัปดาห์ : 8 คน × 5 ชม. = 40 ชม.                  │
  │  ค่าแรงเฉลี่ย          : 500 บาท/ชม.                            │
  │  ต้นทุนสูญเสีย/สัปดาห์: 40 × 500 = 20,000 บาท                 │
  │  ต้นทุนสูญเสีย/ปี     : 20,000 × 48 สัปดาห์ = 960,000 บาท     │
  ├─────────────────────────────────────────────────────────────────┤
  │  หลังใช้ Jira                                                   │
  │  ─────────────────────────────────────────────────────         │
  │  ค่า Jira Standard     : $8.15/user/month × 8 = $65.2/month   │
  │  ≈ 2,300 บาท/เดือน    ≈ 27,600 บาท/ปี                         │
  │  เวลาสูญเสียลดลง 70%  : ประหยัดได้ 672,000 บาท/ปี             │
  ├─────────────────────────────────────────────────────────────────┤
  │  ROI = (672,000 - 27,600) / 27,600 × 100 = 2,336%             │
  │  💡 ทุก 1 บาทที่ลงทุนใน Jira → ได้คืนมา ~23 บาท               │
  └─────────────────────────────────────────────────────────────────┘
```

## 4. Jira Software — เชิงลึก <a name="4"></a>

> 💡 **Jira Software** พัฒนาโดย Atlassian เป็น Issue & Project Tracking Tool ที่ทรงพลังที่สุดสำหรับ Agile Teams รองรับทั้ง Scrum และ Kanban พร้อม Reporting ที่ครบครัน

### 4.0 ทำความรู้จัก Jira ใน 60 วินาที

```
  ⏱️ JIRA IN 60 SECONDS — แนวคิดหลักที่ต้องรู้ก่อน

  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                 │
  │  JIRA = ระบบติดตามงาน ที่ออกแบบมาสำหรับ Software Teams        │
  │                                                                 │
  │  งานแต่ละชิ้น = "Issue"  (คล้าย Ticket หรือ Card)               │
  │  กลุ่มงาน     = "Project"                                        │
  │  ช่วงเวลา     = "Sprint"  (ปกติ 2 สัปดาห์)                       │
  │  รายการงาน    = "Backlog"                                        │
  │                                                                 │
  │  กระบวนการทำงานพื้นฐาน:                                          │
  │                                                                 │
  │  1. PO สร้าง Issues เก็บใน Backlog                              │
  │  2. ทีมเลือก Issues ใส่ Sprint (Sprint Planning)                │
  │  3. Dev ทำงานและอัปเดต Status บน Board                          │
  │  4. Sprint สิ้นสุด → Review → เริ่ม Sprint ใหม่                │
  │  5. ดู Reports เพื่อปรับปรุงการทำงาน                             │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

### 4.8 JQL — Jira Query Language

> 💡 **JQL** เปรียบเหมือน **SQL แต่สำหรับ Jira Issues** — ใช้ค้นหา Filter และ สร้าง Report ที่ซับซ้อนได้อย่างแม่นยำ

```
  🔍 JQL SYNTAX พื้นฐาน

  ┌─────────────────────────────────────────────────────────────────┐
  │  [field] [operator] [value] AND/OR [field] [operator] [value]  │
  │                                                                 │
  │  ตัวอย่างง่าย ๆ :                                               │
  │  assignee = alice                                               │
  │  → หา Issues ทั้งหมดที่ Assign ให้ alice                        │
  │                                                                 │
  │  status = "In Progress" AND priority = High                     │
  │  → หา Issues ที่กำลังทำอยู่ และ Priority สูง                    │
  └─────────────────────────────────────────────────────────────────┘
```

```
  📚 JQL QUERIES ที่ใช้บ่อยในชีวิตจริง

  ─────────────────────────────────────────────────────────────────
  🔸 งานของฉันใน Sprint นี้ (ใช้ทุกวัน)
  assignee = currentUser()
  AND sprint in openSprints()
  AND status != Done
  ORDER BY priority DESC

  ─────────────────────────────────────────────────────────────────
  🔸 Bug ที่ยังไม่แก้ Priority สูง (สำหรับ Bug Triage)
  issuetype = Bug
  AND status != Done
  AND priority in (High, Critical)
  ORDER BY created ASC

  ─────────────────────────────────────────────────────────────────
  🔸 งานที่ค้างใน Code Review นานกว่า 2 วัน (Bottleneck Detector)
  status = "Code Review"
  AND status changed to "Code Review" before -2d
  ORDER BY updated ASC

  ─────────────────────────────────────────────────────────────────
  🔸 ทุก Issue ใน Sprint ที่ผ่านมา (สำหรับ Retrospective)
  project = PROJ
  AND sprint = "Sprint 6"
  ORDER BY status ASC

  ─────────────────────────────────────────────────────────────────
  🔸 Issues ที่ยังไม่มีคนรับ (งานตกหล่น!)
  project = PROJ
  AND assignee is EMPTY
  AND status != Done
  ORDER BY priority DESC

  ─────────────────────────────────────────────────────────────────
  🔸 Release Note สำหรับเดือนนี้
  project = PROJ
  AND status = Done
  AND issuetype in (Story, Bug)
  AND resolutiondate >= startOfMonth()
  ORDER BY resolutiondate ASC
```

```
  📖 JQL Reference — Operators & Functions

  ┌──────────────┬────────────────────────────────────────────────┐
  │  Operators   │  ความหมายและตัวอย่าง                            │
  ├──────────────┼────────────────────────────────────────────────┤
  │  =           │  เท่ากับ:  assignee = "alice"                  │
  │  !=          │  ไม่เท่า:  status != Done                      │
  │  in          │  อยู่ใน:   priority in (High, Critical)        │
  │  not in      │  ไม่อยู่ใน: status not in (Done, Closed)       │
  │  is EMPTY    │  ว่างเปล่า: assignee is EMPTY                  │
  │  ~           │  มีคำนี้:  summary ~ "login"                   │
  │  >  <        │  มากกว่า/น้อยกว่า: created > -7d               │
  │  changed to  │  เคย Transition: status changed to "Done"      │
```

## 7. การออกแบบ Workflow ที่ดี <a name="7"></a>

> 💡 **Workflow** คือเส้นทางที่ Work Item เดินทางจากเริ่มต้นจนเสร็จ การออกแบบที่ดีต้อง **สะท้อนความเป็นจริง** ของทีม ไม่ใช่ Copy มาจาก Template แล้วไม่มีใครทำตาม

### 7.1 หลักการออกแบบ Workflow ที่ดี

```
╔══════════════════════════════════════════════════════════════════╗
║            ✏️  5 หลักการออกแบบ Workflow ที่ดี                    ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  1️⃣  REFLECT REALITY ── สะท้อนความจริง                          ║
║     ออกแบบตาม Process ที่ทีมทำจริง ไม่ใช่ที่ "ควรจะเป็น"        ║
║     ถามทีมว่า "งานชิ้นหนึ่งผ่านขั้นตอนอะไรบ้างก่อนเสร็จ?"        ║
║                                                                  ║
║  2️⃣  FEWER IS BETTER ── น้อยดีกว่า                              ║
║     เริ่มจาก 3-5 Status แล้วเพิ่มเมื่อจำเป็นจริง ๆ              ║
║     Status ที่ไม่มีงานอยู่เลยควรลบออก                             ║
║                                                                  ║
║  3️⃣  CLEAR CRITERIA ── เกณฑ์ชัดเจน                              ║
║     ทุก Status ต้องตอบได้ว่า "เข้าเมื่อไหร่" และ "ออกเมื่อไหร่"   ║
║     เขียน Definition of Done ให้ชัดเจน                           ║
║                                                                  ║
║  4️⃣  VISUALIZE HANDOFFS ── เห็นจุดส่งต่อ                        ║
║     จุดที่งานส่งจาก Dev → QA หรือ Dev → Reviewer                ║
║     ควรเป็น Status ที่ชัดเจนบน Board                             ║
║                                                                  ║
║  5️⃣  REVIEW & ADAPT ── ทบทวนและปรับปรุง                         ║
║     ทบทวน Workflow ใน Sprint Retrospective ทุก 1-2 เดือน         ║
║     Workflow ที่ดีจะวิวัฒนาการตามทีม                              ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

### 7.2 Workflow Templates ตามขนาดและประเภททีม

#### Template A: Starter Workflow (ทีม 2-5 คน, เพิ่งเริ่ม)

```
  [TO DO] ──────────► [IN PROGRESS] ──────────► [DONE]

  ─────────────────────────────────────────────────────────────────
  Entry/Exit Criteria:

  ┌───────────────┬────────────────────────┬──────────────────────┐
  │  Status       │  เข้าเมื่อ              │  ออกเมื่อ            │
  ├───────────────┼────────────────────────┼──────────────────────┤
  │ TO DO         │ Sprint Planning        │ Dev เริ่มทำงาน        │
  │               │ เลือก Issue เข้า Sprint│                      │
  ├───────────────┼────────────────────────┼──────────────────────┤
  │ IN PROGRESS   │ Dev เริ่ม Code         │ งานเสร็จ, ทดสอบแล้ว  │
  │               │ สร้าง Branch แล้ว      │ Self-review ผ่าน      │
  ├───────────────┼────────────────────────┼──────────────────────┤
  │ DONE          │ ทำเสร็จและ Test เอง    │ Sprint Complete       │
  └───────────────┴────────────────────────┴──────────────────────┘

  ✅ เหมาะกับ: Side Project, Prototype, ทีมใหม่ที่เพิ่งเริ่ม Tracking
```

#### Template B: Standard Development Workflow (ทีม 5-15 คน, มี Code Review + QA)

```
  [BACKLOG] → [TO DO] → [IN PROGRESS] → [CODE REVIEW] → [TESTING] → [DONE]

  ─────────────────────────────────────────────────────────────────
  Entry/Exit Criteria ละเอียด:

  ┌────────────────┬───────────────────────────┬────────────────────┐
  │  Status        │  Entry Criteria            │  Exit Criteria     │
  ├────────────────┼───────────────────────────┼────────────────────┤
  │ TO DO          │ Story มี AC ครบ,           │ Dev เริ่มทำงาน     │
  │                │ Estimate แล้ว              │                    │
  ├────────────────┼───────────────────────────┼────────────────────┤
  │ IN PROGRESS    │ Dev เริ่ม Code,            │ PR สร้างแล้ว,      │
  │                │ สร้าง Branch แล้ว          │ CI ผ่าน,           │
  │                │                           │ Self-review แล้ว   │
  ├────────────────┼───────────────────────────┼────────────────────┤
  │ CODE REVIEW    │ PR Created + CI ผ่าน       │ 1+ Approval,       │
  │                │                           │ ทุก Comment Resolve │
  │                │                           │ Merge สำเร็จ        │
  ├────────────────┼───────────────────────────┼────────────────────┤
  │ TESTING        │ Deploy ขึ้น Staging ได้    │ QA ยืนยัน AC ทุกข้อ │
  │                │ Test Plan เตรียมพร้อม      │ ไม่มี Critical Bug  │
  ├────────────────┼───────────────────────────┼────────────────────┤
  │ DONE           │ ผ่านทุก Exit Criteria     │ Sprint Complete     │
  └────────────────┴───────────────────────────┴────────────────────┘
```

```
  ┌─────────────────────────────────────────────────────────────────┐
```

```
  │  ─────────────────────────────────────────────────────         │
  │  ✅ Build: npm run build          (45 sec)                     │
  │  ✅ Test:  142/142 tests passed   (2 min 30 sec)               │
  │  ✅ Lint:  No errors found        (15 sec)                     │
  │  ✅ Deploy to Staging             (1 min 20 sec)               │
  └──────────────────────────┬─────────────────────────────────────┘
                             │
          ┌──────────────────┼───────────────────┐
          │                  │                   │
          ▼                  ▼                   ▼
  ┌──────────────┐  ┌───────────────┐  ┌────────────────┐
  │  Jira Update │  │  Slack Notify │  │  PR Auto-check │
  │  ────────────│  │  ─────────────│  │  ──────────────│
  │  PROJ-142    │  │  #dev-alerts  │  │  ✅ CI Passed  │
  │  Status →    │  │  "✅ Build OK │  │  Auto-request  │
  │  TESTING     │  │  PROJ-142     │  │  Review from   │
  │              │  │  Staging live"│  │  @lead-dev     │
  └──────────────┘  └───────────────┘  └────────────────┘

  ─────────────────────────────────────────────────────────────────
  ตัวอย่าง GitHub Actions ที่อัปเดต Jira อัตโนมัติ:

  # .github/workflows/update-jira.yml
  name: "Auto Update Jira on Deploy"
  on:
    push:
      branches: [main]

  jobs:
    update-jira:
      runs-on: ubuntu-latest
      steps:
        - name: "Extract Jira Issue Keys from Commits"
          id: extract
          run: |
            ISSUES=$(git log --oneline -10 \
              | grep -oE 'PROJ-[0-9]+' | sort -u | tr '\n' ',')
            echo "issues=$ISSUES" >> $GITHUB_OUTPUT

        - name: "Transition Issues to Done"
          uses: atlassian/gajira-transition@v3
          with:
            issue: ${{ steps.extract.outputs.issues }}
            transition: "Done"
            token: ${{ secrets.JIRA_API_TOKEN }}

        - name: "Notify Slack"
          uses: 8398a7/action-slack@v3
          with:
            status: ${{ job.status }}
            text: |
              🚀 *Deployed to Production*
              Issues: ${{ steps.extract.outputs.issues }}
              Branch: ${{ github.ref_name }}
          env:
            SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}
```

## 9. Best Practices & Anti-Patterns <a name="9"></a>

### 9.1 Best Practices — สิ่งที่ทีมที่ดีทำ

```
╔══════════════════════════════════════════════════════════════════╗
║           ✅ BEST PRACTICES สำหรับ JIRA                          ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  📝 ISSUE HYGIENE — เขียน Issue ให้มีคุณภาพ                      ║
║  ──────────────────────────────────────────────────────────     ║
║  ✅ ทุก Issue มี Description ที่อ่านแล้วเข้าใจทันที             ║
║  ✅ Bug Report มี Steps to Reproduce + Environment เสมอ         ║
║  ✅ Story มี Acceptance Criteria ที่ Testable                    ║
║  ✅ Issue มี Assignee ก่อนย้ายเข้า In Progress                   ║
║  ✅ Archive/Delete Issues ที่ไม่ทำแน่ ๆ                          ║
║                                                                  ║
║  🏃 SPRINT MANAGEMENT — จัดการ Sprint ให้มีประสิทธิภาพ           ║
║  ──────────────────────────────────────────────────────────     ║
║  ✅ ทุก Sprint มี Sprint Goal 1 ประโยคที่ชัดเจน                  ║
║  ✅ เลือกงานตาม Velocity จริง ไม่ Overload                       ║
║  ✅ ไม่เพิ่มงานกลาง Sprint ยกเว้นกรณีพิเศษ                       ║
║  ✅ ทำ Backlog Refinement สม่ำเสมอ (สัปดาห์ละครั้ง)              ║
║                                                                  ║
║  📊 REPORTING — ใช้ข้อมูลมาตัดสินใจ                              ║
║  ──────────────────────────────────────────────────────────     ║
║  ✅ Review Burndown Chart ทุกวัน Stand-up                       ║
║  ✅ ดู Velocity Chart ทุก Sprint เพื่อ Plan ครั้งถัดไป           ║
║  ✅ ทำ Sprint Report ก่อน Retrospective ทุกครั้ง                 ║
║  ✅ Share Dashboard กับ Stakeholders แทนการ Email รายงาน         ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝

╔══════════════════════════════════════════════════════════════════╗
║           ✅ BEST PRACTICES สำหรับ TRELLO                        ║
╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
║  📋 BOARD DESIGN — ออกแบบให้ใช้งานได้จริง                        ║
║  ──────────────────────────────────────────────────────────     ║
║  ✅ List ชื่อต้องบอก Action: "🔨 In Development" ดีกว่า "Doing" ║
```

### 9.2 Anti-Patterns ที่พบบ่อยและวิธีแก้

```
  ⚠️ ANTI-PATTERN 1: "The Abandoned Board" (บอร์ดร้าง)
  ─────────────────────────────────────────────────────────────────

  🔍 อาการ:
  → Board สวยงามในวันแรก แต่ไม่มีใข้แตะมา 3 สัปดาห์
  → Dev Update งานบ้างไม่บ้างตามอารมณ์
  → Stand-up ก็ยังต้องถามกัน Board ไม่มีประโยชน์

  💸 ผลเสีย:
  → Board ไม่ Reflect ความเป็นจริง
  → ทีมไม่เชื่อ Board และเลิกใช้ในที่สุด

  ✅ วิธีแก้:
  → "Jira/Trello First" Policy: ก่อน Update วิธีอื่น Update Board ก่อน
  → Review Board ทุกวัน Stand-up (5 นาที)
  → Lead/SM ตรวจ Stale Issues ทุกสัปดาห์

  ─────────────────────────────────────────────────────────────────
  ⚠️ ANTI-PATTERN 2: "The Graveyard Backlog" (Backlog สุสาน)
  ─────────────────────────────────────────────────────────────────

  🔍 อาการ:
  → Backlog มี 300+ Issues ส่วนใหญ่อายุมากกว่า 1 ปี
  → ไม่มีใครรู้ว่าอันไหนยังต้องทำอยู่

  💸 ผลเสีย:
  → Sprint Planning ใช้เวลาเป็นชั่วโมงเพราะต้องคัดกรอง
  → งานสำคัญหายไปในกองขยะ Issue เก่า ๆ

  ✅ วิธีแก้:
  → Backlog Grooming เดือนละครั้ง: Archive/Delete ที่ไม่จำเป็น
  → Keep Backlog ≤ 3 Sprints worth of work
  → Issues ที่ไม่ได้ Touch 3 เดือน → Auto-archive

  ─────────────────────────────────────────────────────────────────
  ⚠️ ANTI-PATTERN 3: "Status Theater" (แสดงละครสถานะ)
  ─────────────────────────────────────────────────────────────────

  🔍 อาการ:
  → ย้าย Card ไป "In Progress" วันแรกของ Sprint ทุกใบ
  → ทั้งที่ยังไม่ได้เริ่มทำจริง ๆ
  → Board ดูดีบน Paper แต่ไม่ตรงความเป็นจริง

  💸 ผลเสีย:
  → Burndown Chart สวยปลอม Management เข้าใจผิด
  → WIP Limit ไม่ทำงาน Bottleneck มองไม่เห็น

  ✅ วิธีแก้:
  → Policy ชัดเจน: "In Progress = กำลังทำอยู่จริง ๆ ตอนนี้"
  → WIP Limit ต้อง Enforce จริง ไม่ใช่แค่ตั้งไว้ให้ดูดี
  → ทำ Written Policy ใน Board Description

  ─────────────────────────────────────────────────────────────────
  ⚠️ ANTI-PATTERN 4: "The Over-engineered Board" (Board ซับซ้อนเกิน)
  ─────────────────────────────────────────────────────────────────

  🔍 อาการ:
  → Workflow มี 15+ Status, Custom Fields มากมาย
  → ทีมใช้เวลา Update Jira มากกว่าเขียนโค้ด

  💸 ผลเสีย:
  → ทีมรู้สึก Tool คือภาระ ไม่ใช่ประโยชน์
  → Onboard คนใหม่ยากมาก ใช้เวลาสอนนาน

  ✅ วิธีแก้:
  → เริ่มต้นด้วย 4-5 Status เสมอ
  → เพิ่ม Complexity เมื่อเจอปัญหาจริง ๆ เท่านั้น
  → ถามก่อนทุกครั้งที่จะเพิ่ม Field: "เอาข้อมูลนี้ไปใช้ทำอะไร?"

  ─────────────────────────────────────────────────────────────────
  ⚠️ ANTI-PATTERN 5: "The Information Silo" (ข้อมูลแยกกองโดด)
  ─────────────────────────────────────────────────────────────────

  🔍 อาการ:
  → Dev ทำงานเสร็จแต่ไม่ Comment ใน Issue
  → ข้อมูลสำคัญอยู่ในหัว หรือใน Chat ส่วนตัว
  → คนอื่นต้องถามซ้ำ ๆ เพราะไม่รู้ว่ามีข้อมูลที่ไหน

  ✅ วิธีแก้:
  → Policy: "ถ้าสำคัญ ต้องอยู่ใน Issue Comment"
  → Decision ทุกข้อที่เกี่ยวกับ Issue ต้อง Log ใน Comment
  → ใช้ @mention ใน Comment แทนการ Chat ส่วนตัว
```

### 10.1 Case Study 1: Startup 4 คน — จาก WhatsApp สู่ Trello

```
  📱 CASE STUDY: HealthTech Startup — 4 คน

  ─────────────────────────────────────────────────────────────────
  บริบท:

  ─────────────────────────────────────────────────────────────────
  สิ่งที่สร้างในวันแรก (ใช้เวลา 1 วัน):

  ┌──────────┐  ┌────────────┐  ┌──────────┐  ┌──────────┐

  │          │  │            │  │ (Max: 2) │  │          │

  ─────────────────────────────────────────────────────────────────

  → ทุกเช้าจันทร์: ทุกคนเปิด Trello ก่อนเช็ค WhatsApp
  → INBOX → THIS WEEK: PM เท่านั้นที่ย้ายได้ (Priority ชัด)
  → Doing WIP ≤ 2: ไม่เริ่มงานใหม่ถ้า Doing ยังเต็ม
  → Done แล้ว → Comment: บอกว่าอะไรเสร็จ ทำอะไรไป

  ─────────────────────────────────────────────────────────────────
  ผลลัพธ์หลัง 1 เดือน:

  ✅ เวลาตาม "สถานะงาน" ลดจาก 3 ชม. → 30 นาที/สัปดาห์
  ✅ ไม่มีงานหายอีก
  ✅ Meeting ลดจาก 5 → 2 ครั้ง/สัปดาห์
  ✅ Feature Delivery เร็วขึ้น ~40%

  ─────────────────────────────────────────────────────────────────
  Lessons Learned:
  💡 "Board ที่ทุกคน Update จริงดีกว่า Board ที่สวยแต่ไม่ใช้"
  💡 กฎ WIP = 2 บังคับทีมให้ Focus ได้จริง
  💡 Habit สำคัญกว่า Feature ของ Tool
```

### 10.2 Case Study 2: Fintech บริษัท 25 คน — ใช้ Jira ครบวงจร


### ✏️ Exercise 1: วิเคราะห์ Issue ที่ดีและไม่ดี

```
  โจทย์: อ่าน Issue ด้านล่าง ระบุปัญหา แล้วเขียนใหม่ให้ดีขึ้น

  ─────────────────────────────────────────────────────────────────
  ❌ Issue ที่ไม่ดี:

  Title:       "แก้ Bug Login"
  Type:        Bug
  Description: "มีปัญหา Login บาง Chrome"
  Assignee:    (ว่าง)
  Priority:    Medium
  Labels:      (ว่าง)

  ─────────────────────────────────────────────────────────────────
  คำถาม:
  1. ระบุปัญหาที่มีใน Issue นี้อย่างน้อย 5 ข้อ
  2. เขียน Issue ที่ดีขึ้นพร้อม:
     - Title ที่ดี (บอก What + Where + Impact)
     - Steps to Reproduce (อย่างน้อย 4 ขั้นตอน)
     - Expected vs Actual Behavior
     - Environment (Browser, OS, Version)
     - Priority ที่เหมาะสมพร้อมเหตุผล

  ─────────────────────────────────────────────────────────────────
  💡 แนวทางเฉลย — ปัญหา 5 ข้อ:
  ❌ Title ไม่บอกว่า Bug คืออะไร, ที่ไหน, กระทบใคร
  ❌ Description ไม่มี Steps to Reproduce
  ❌ ไม่มี Assignee — ไม่มีใครรับผิดชอบ
  ❌ "บาง Chrome" ไม่ระบุ Version ที่เกิดปัญหา
  ❌ ไม่มี Acceptance Criteria ไม่รู้ว่าเมื่อไหร่ถือว่าแก้เสร็จ

  💡 ตัวอย่าง Issue ที่ดี:
  Title: "🐛 [Login] Chrome 119+ ไม่รับ Session Cookie ทำให้วน Login Loop"

  Steps to Reproduce:
  1. เปิด Chrome เวอร์ชัน 119.0.6045.105 ขึ้นไป
  2. ไปที่ https://app.example.com/login
  3. กรอก Email: test@example.com | Password: Test1234!
  4. กด "Sign In"

  Expected: Redirect ไปหน้า Dashboard (/dashboard)
  Actual: วนกลับมาหน้า Login ซ้ำ ไม่มี Error Message

  Environment:
  OS: Windows 11 Pro / macOS 14 Sonoma
  Browser: Chrome 119.0.6045.105 (ขึ้นไป)
  Note: Firefox และ Safari ใช้งานได้ปกติ

  Priority: 🔴 High — กระทบผู้ใช้ Chrome ทุกราย (~70% ของ Users)
```

### ✏️ Exercise 3: JQL Practice

```
  โจทย์: เขียน JQL Query สำหรับแต่ละ Scenario

  Project Key: SHOP  |  Team: Alice (FE), Bob (BE), Charlie (QA)

  ─────────────────────────────────────────────────────────────────
  1. ดูงานทั้งหมดที่ Charlie รับผิดชอบ ยังไม่เสร็จ เรียงตาม Priority

  2. ดู Bug ทุกตัวที่ Priority สูงกว่า Medium ใน Sprint ปัจจุบัน

  3. ดู Issues ทั้งหมดที่อยู่ใน "Code Review" นานกว่า 1 วัน
     (สัญญาณ Bottleneck)

  4. ดูทุก Issues ที่ไม่มี Assignee ยังไม่เสร็จ
     (งานที่ตกหล่น ไม่มีใครรับ)

  5. สร้าง Query สำหรับ Release Note:
     ดู Stories และ Bugs ที่ Done ในเดือนนี้

  ─────────────────────────────────────────────────────────────────
  ✅ เฉลย:

  1. assignee = charlie
     AND status != Done
     ORDER BY priority DESC, created ASC

  2. issuetype = Bug
     AND priority in (High, Critical)
     AND sprint in openSprints()
     ORDER BY priority DESC

  3. status = "Code Review"
     AND status changed to "Code Review" before -1d
     ORDER BY updated ASC

  4. project = SHOP
     AND assignee is EMPTY
     AND status != Done
     ORDER BY priority DESC

  5. project = SHOP
     AND issuetype in (Story, Bug)
     AND status = Done
     AND resolutiondate >= startOfMonth()
     AND resolutiondate <= endOfMonth()
     ORDER BY resolutiondate ASC
```

```

  ─────────────────────────────────────────────────────────────────
  บริษัท: SecurePay (FinTech — ระบบชำระเงิน)

  • 1 Product Owner
  • 1 Scrum Master

  ⚠️ ทุก Feature ต้องผ่าน Security Review ก่อน QA Testing

  ⚠️ Critical Bug ต้องแก้ภายใน 4 ชั่วโมงหลังรับรายงาน

  1. Workflow Diagram สำหรับ Feature Stories

  Feature Workflow (แนะนำให้มี Security Stage):
  TO DO → DEV → CODE REVIEW → SECURITY REVIEW → QA TESTING
       → READY TO DEPLOY → PRODUCTION ✅

  Critical Bug Fast Track:
  NEW → CONFIRMED (< 30 min) → IN PROGRESS → FIX REVIEW
     → VERIFY → HOTFIX DEPLOYED (< 4 ชม. รวม)
```
