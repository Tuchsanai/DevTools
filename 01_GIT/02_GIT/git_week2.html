อ้างอิงสไลด์หน้า 5–15 · Basic Git Usage

มี 2 วิธี ในการเริ่มต้น repository บนเครื่องของเรา:

🐙 เริ่มจาก GitHub แล้ว Clone

สร้าง Repo บน GitHub ก่อน แล้วดึงสำเนาทั้งหมดลงมาบนเครื่อง

# 1) สร้าง repo บน github.com
# 2) คัดลอกมาทั้งก้อน

เหมาะเมื่อโปรเจกต์เริ่มที่ GitHub

# เกิดโฟลเดอร์ซ่อน .git ขึ้นมา

สามพื้นที่ของ Git: Add & Commit

อ้างอิงสไลด์หน้า 16–36 · Add and Commit

หัวใจของ Git คือการเดินทางของไฟล์ผ่าน 3 พื้นที่ (three areas):

📝
Working Directory
ที่เราแก้ไฟล์
→git add
📦
Staging Area
(index)
→git commit
🗄️
Repository
(.git)
  • Working Directory — โฟลเดอร์ที่เราแก้ไฟล์จริง (program.py, index.html, style.css)
  • Staging Area — "ห้องเตรียมตัว" รวบรวมไฟล์ที่จะ commit ในรอบนี้ ด้วย git add
  • Repository — ที่เก็บประวัติถาวร เมื่อ git commit แล้ว snapshot จะถูกบันทึกไว้

เดินตามตัวอย่างทีละสเต็ป

1
stage ไฟล์เดียว — ย้าย program.py เข้า Staging Area 
git add program.py
2
commit สิ่งที่ stage ไว้ลง Repository พร้อมข้อความอธิบาย 
git commit -m "python code"
Staging Area ว่างลง และ Repository ได้ snapshot ของ program.py
3
stage ทุกไฟล์ที่เหลือ ด้วยจุด (.) = ทุกไฟล์ใน working directory 
git add .
ตอนนี้ index.html และ style.css เข้า Staging Area
4
commit รอบที่สอง 
git commit -m "site files"
Repository มี 2 commit แล้ว: "python code" และ "site files"
5
push commit ทั้งหมดขึ้น remote (GitHub) → ดูรายละเอียดในหัวข้อถัดไป 
git push
⚠️ ลำดับสำคัญ: ไฟล์ต้องผ่าน add → commit เสมอ การแก้ไฟล์เฉย ๆ ไม่ได้แปลว่า Git บันทึกให้ ต้อง git add เพื่อ stage ก่อน แล้วจึง git commit

กายวิภาคของ Commit (Commit Anatomy)

อ้างอิงสไลด์หน้า 37 · โครงสร้างภายในของ commit

แต่ละ commit มี รหัส (hash) ของตัวเอง และชี้ไปยัง commit แม่ (parent) ทำให้เกิดเป็น ห่วงโซ่ของประวัติ ที่ย้อนกลับได้

add banner
hash: 171615…ffd8765
parent: 236ff65…
message: add banner
←
fix typo
hash: 236ff65…df35d897
parent: 987fac6…
message: fix typo
←
remove x…
hash: 987fac6…6ce7dd765
parent: none
message: remove x…

ลูกศร ← ชี้จากแต่ละ commit กลับไปยัง parent ของมัน (commit ใหม่สุดอยู่ซ้าย) — ตรงกับสไลด์หน้า 37

🔑

hash

ลายนิ้วมือเฉพาะตัวของ commit (SHA) ใช้ระบุ commit ได้ไม่ซ้ำกัน

🔗

parent

ตัวชี้ไปยัง commit ก่อนหน้า — commit แรกสุดมี parent เป็น none

💬

message

ข้อความที่เราเขียนตอน -m อธิบายว่า commit นี้ทำอะไร

Push & Remote Branches

อ้างอิงสไลด์หน้า 38–49 · Push and Remote Branches

ตรวจสอบ remote ที่มีอยู่

git remote -v
  • ถ้ารันบน repo ที่ clone มา จะเห็น URL ของ remote (เช่น URL ของ GitHub)
  • ถ้า ยังไม่เชื่อมต่อ remote ใด ก็จะ ไม่เห็น URL

เพิ่ม remote branch (origin)

เราบอก Git ให้เพิ่ม remote ด้วยรูปแบบ git remote add <name> <url> โดย ตามธรรมเนียม เราจะตั้งชื่อ remote นี้ว่า origin

# รูปแบบทั่วไป
git remote add name https://url.git

# ตามธรรมเนียมใช้ชื่อ origin (แทน url ด้วย .git URL ของ repo เรา)
git remote add origin https://github.com/user/repo.git
📝 เผื่ออนาคต (ไม่ได้ใช้ตอนนี้): เปลี่ยนชื่อ/ลบ remote ได้ด้วย 
git remote rename <old> <new>
git remote remove <name>

Push ขึ้น remote ที่ชื่อ origin

เมื่อเชื่อมต่อกับ remote บน GitHub แล้ว สั่ง push โค้ดขึ้นไปยัง remote ที่ชื่อ origin (ที่ branch main/master) ได้ด้วย

git push -u origin main      # หรือ master ถ้า branch หลักชื่อ master

ตัวเลือก -u (upstream) ทำให้ครั้งต่อ ๆ ไปสั่งแค่ git push สั้น ๆ ก็พอ

📌 origin = ชื่อ remote ไม่ใช่ชื่อ branch: origin คือชื่อที่เราตั้งให้ remote repository ส่วน main/master คือชื่อ branch — คำสั่ง git push origin main จึงหมายถึง "push branch main ไปยัง remote ชื่อ origin"
⚠️ master → main: GitHub เปลี่ยน default branch ของ repo ที่สร้างใหม่ จาก master เป็น main อย่างเป็นทางการแล้ว (repo เก่าที่ยังใช้ master ไม่ได้ถูกเปลี่ยนย้อนหลัง) คำสั่งที่ GitHub แนะนำให้เปลี่ยนชื่อ branch ปัจจุบันเป็น main คือ 
git branch -M main

โฟลเดอร์ในเครื่อง → GitHub โดยไม่ต้อง Clone

ถ้าเรามีโฟลเดอร์อยู่ในเครื่อง (LOCAL DESKTOP) อยู่แล้ว และอยากผูกกับ repo บน GitHub โดยไม่ clone ใหม่ — ต้องระวังเรื่องชื่อ branch ให้ ตรงกัน (main ↔ main) ถ้าชื่อไม่ตรง (master ↔ main) แล้ว push ชื่อ branch ตรง ๆ เช่น git push -u origin master Git จะไป สร้าง branch master แยกอีกตัวบน remote คนละตัวกับ main กลายเป็น improper reference — มี branch หลักสองสาขาไม่ตรงกัน (ส่วน git push เปล่า ๆ ที่ยังไม่ตั้ง upstream จะ ฟ้อง error แทน) จึงควร git branch -M main ให้ชื่อ branch ในเครื่องตรงกับ remote ก่อน

A Closer Look at Cloning & Remote-Tracking Branches

อ้างอิงสไลด์หน้า 50–62 · Remote-Tracking Branches & Detached HEAD

เมื่อ git clone เราจะได้ branch 2 แบบ บนเครื่อง:

🟢 master (local branch)

ตัวชี้ branch ปกติ ของเรา — เราเลื่อนมันเองได้ ทุกครั้งที่ commit ใหม่ ตัวชี้ master จะขยับไปข้างหน้า

🔖 origin/master (remote-tracking)

"Remote-Tracking Branch" — เป็น บุ๊กมาร์ก ที่ชี้ไปยังสถานะล่าสุดของ master บน remote ที่เรารู้ เราขยับมันเองไม่ได้ มันจะอัปเดตเมื่อ sync กับ remote เท่านั้น

ดู remote branches ที่เครื่องรู้จัก

git branch -r

เมื่อ commit ในเครื่อง — ตัวชี้ขยับไม่พร้อมกัน

1
เริ่มต้น: master และ origin/master ชี้ commit เดียวกัน
2
commit ใหม่ในเครื่อง → master ขยับไปข้างหน้า แต่ origin/master อยู่ที่เดิม
3
commit อีกครั้ง → master ขยับอีก ส่วน origin/master ยังไม่ขยับ (จนกว่าจะ push)

เมื่อรัน git status Git จะบอกว่าเรา "ahead" อยู่กี่ commit:

git status
On branch master
Your branch is ahead of 'origin/master' by 2 commits.
  (use "git push" to publish your local commits)

Detached HEAD — ไม่ต้องตกใจ

เราสามารถ checkout ไปดูสถานะ ณ remote pointer หรือ commit เก่าได้ เช่น "ตอน clone มาครั้งแรกหน้าตาเป็นยังไงนะ?"

git checkout origin/master
Note: switching to 'origin/master'.
You are in 'detached HEAD' state. You can look around, make
experimental changes and commit them, and you can discard
any commits you make in this state...
😌 Detached HEAD ไม่ใช่ error! มันแค่หมายความว่า HEAD ชี้ตรงไปที่ commit ใด commit หนึ่ง (ไม่ใช่ปลายของ branch) เรา "มองดูได้ ทดลองได้" และกลับมา branch เดิมได้เสมอ
⚠️ ระวังตอน commit ขณะ detached: commit ที่ทำระหว่าง detached HEAD จะกลายเป็น unreachable เมื่อ checkout กลับไป branch อื่น (ยังกู้คืนได้ชั่วคราวผ่าน git reflog ก่อนถูก garbage-collect) ถ้าอยากเก็บถาวร ต้อง สร้าง branch ก่อนกลับ เช่น git switch -c new-branch (หรือ git branch new-branch <hash>)

Clone แล้ว ≠ มีทุก branch ใน workspace

เมื่อ clone มา เราได้ ข้อมูลและประวัติ Git ทั้งหมด ณ ขณะนั้น แต่ ไม่ได้แปลว่าทุก branch อยู่ใน workspace เช่น remote มี branch ชื่อ puppies แต่พอรัน git branch กลับเห็นแค่ master:

git branch
* master

เป็นเรื่องปกติ — branch อื่น ๆ ของ remote ยังอยู่ใน Git history แต่ยังไม่ถูกสร้างเป็น local branch จนกว่าเราจะ checkout มัน

Git Log & Diff — ดูประวัติและความเปลี่ยนแปลง

อ้างอิงสไลด์หน้า 63–64 (Git Log) · git diff จากภาพรวมคำสั่งสไลด์หน้า 2

ก่อนจะใช้ git fetch และ git pull มารู้จัก git log กันก่อน — มันแสดง รายการ commit ทั้งหมด ที่เคยทำใน repo พร้อม hash, message และ metadata มองว่ามันคือ "ประวัติของ repo" ได้เลย

# ประวัติแบบเต็ม (hash, author, date, message)
git log

# แบบกระชับ บรรทัดเดียวต่อ commit
git log --oneline

# ดู log ของทุก branch ไม่ใช่แค่ branch ปัจจุบัน
git log --all --oneline
🔎 ใช้ git log --oneline เพื่อหา commit hash สั้น ๆ ที่จะนำไปใช้กับ git checkout <hash> เพื่อย้อนไปดูสถานะเก่าได้

git diff — ดูว่าแก้อะไรไปบ้าง

git diff แสดง ความแตกต่าง (changes) ของไฟล์ที่แก้ไปแล้วแต่ ยังไม่ได้ stage (ยังไม่ได้ git add) บรรทัดที่ เพิ่ม นำหน้าด้วย + บรรทัดที่ ลบ นำหน้าด้วย -

git diff
diff --git a/file1.txt b/file1.txt
@@ -1 +1,2 @@
 Content for file 1
+Additional content for file 1     # บรรทัดที่เพิ่มเข้ามา

Fetch vs Pull

อ้างอิงสไลด์หน้า 65–76 · Fetch and Pull

มี 2 วิธี ในการนำการเปลี่ยนแปลงจาก remote branch (เช่นบน GitHub) ลงมา: git fetch และ git pull ภาพรวมของวงจรทั้งหมดเป็นดังนี้:

📝
Workspace
(Working Dir)
→git add
📦
Staging
(index)
→git commit
💻
Local
Repository
→git push
☁️
Remote
Repository

ขากลับ: git fetch และ git pull นำการเปลี่ยนแปลงจาก Remote กลับลงมา

📂 pull ดึงสถานะล่าสุดลงมาที่ Working Directory: git pull นำ commit และสถานะไฟล์ล่าสุดจาก remote ลงมารวมกับ local — ไฟล์ที่ถูก track (เช่น index.html, style.css) จะถูกซิงก์ให้ตรงกับ remote (อ้างอิงสไลด์หน้า 34–36)
สูตรสำคัญ: git pull = git fetch + git merge
  • git fetch → อัปเดต remote-tracking branch (เช่น origin/main) ด้วยการเปลี่ยนแปลงล่าสุดจาก remote
  • git merge → นำการเปลี่ยนแปลงจาก remote-tracking branch มารวมเข้ากับ branch ปัจจุบันของเรา

เปรียบเทียบ git fetch กับ git pull

ประเด็นgit fetchgit pull
ดึงการเปลี่ยนแปลงจาก remote branch✅ ใช่✅ ใช่
อัปเดต remote-tracking branch✅ อัปเดต✅ อัปเดต
merge เข้า HEAD/branch ปัจจุบัน❌ ไม่ merge✅ merge ให้อัตโนมัติ
โอกาสเกิด merge conflictไม่เกิด (ปลอดภัย)อาจเกิด conflict ได้
ความปลอดภัยทำได้ทุกเมื่อ (safe)ไม่แนะนำถ้ายังมีไฟล์ที่ยังไม่ commit
✅ git fetch — ปลอดภัยเสมอ เพราะแค่ดึงข้อมูลมาเก็บไว้ที่ remote-tracking branch ยังไม่แตะ branch ที่เรากำลังทำงานอยู่ เราตรวจสอบก่อนค่อย merge เองได้
⚠️ git pull — สะดวกกว่า (ดึง+merge ในคำสั่งเดียว) แต่ อาจเกิด merge conflict และ ไม่แนะนำให้ใช้ขณะที่ยังมีการเปลี่ยนแปลงที่ยังไม่ได้ commit (uncommitted changes)

LAB 01

add · commit · status (Local)

🎯 ฝึกวงจรพื้นฐาน: ติดตามการเปลี่ยนแปลงของ 3 ไฟล์ ผ่าน 3 รอบการแก้ไข บนเครื่องตัวเอง (ยังไม่มี remote)

git initgit statusgit addgit commitgit log
# Step 1: สร้าง repo + commit แรก
mkdir git-lab && cd git-lab
git init
echo "Content for file 1" > file1.txt
echo "Content for file 2" > file2.txt
echo "Content for file 3" > file3.txt
git status                       # เห็น untracked files
git add file1.txt file2.txt file3.txt
git commit -m "Initial commit with three files"
git status && git log            # ✔ ตรวจสอบหลัง commit ทุกครั้ง

# Step 2: แก้ไฟล์แล้ว commit รอบสอง
echo "Additional content for file 1" >> file1.txt
echo "Additional content for file 2" >> file2.txt
git status                       # เห็น modified files
git add file1.txt file2.txt
git commit -m "Updated file1 and file2"
git status && git log            # ✔ ตรวจสอบอีกครั้ง

# Step 3: แก้ file3 แล้ว commit รอบสุดท้าย
echo "Additional content for file 3" >> file3.txt
git add file3.txt
git commit -m "Final update to file3"
git status && git log            # ✔ ตรวจสอบรอบสุดท้าย
LAB 02

push ขึ้น Remote + แก้ไข 3 รอบ

🎯 ผูก local repo เข้ากับ remote บน GitHub/GitLab/Bitbucket แล้ว push การเปลี่ยนแปลง 3 รอบ

git initgit remote add origingit push -ugit push
# 1) สร้าง local repo + commit แรก
mkdir lab-git && cd lab-git
git init
echo "Initial content for file 1" > file1.txt
echo "Initial content for file 2" > file2.txt
echo "Initial content for file 3" > file3.txt
git add file1.txt file2.txt file3.txt
git commit -m "Initial commit with three files"

# 2) สร้าง remote repo บน GitHub แล้วผูก origin + push
git remote add origin REMOTE-URL
git push -u origin master   # Git รุ่นใหม่ default = main; ถ้า branch ชื่อ main ใช้ origin main

# 3) แก้ไฟล์ + commit + push ซ้ำ 3 รอบ (ใบงานให้แก้ทั้ง file1/file2/file3 ทุกรอบ)
echo "Change 1 for file 1" >> file1.txt    # ตัวอย่างย่อ — ของจริงแก้ครบทั้ง 3 ไฟล์
git add .
git commit -m "First set of changes"
git push origin master
# ... ทำซ้ำเป็น "Second set of changes", "Third set of changes"
LAB 03

fetch + log (จำลองเพื่อนร่วมทีม)

🎯 จำลองสถานการณ์: เพื่อน clone ไปแก้แล้ว push — เรา fetch การเปลี่ยนแปลง ดู log แล้ว merge เข้ามา

# ── ใน main-repo: ตั้งต้น + push ขึ้น remote (branch main) ──
mkdir main-repo && cd main-repo
git init

echo "Initial content for file 2" > file2.txt
echo "Initial content for file 3" > file3.txt

git remote add origin [REMOTE-URL]
git push -u origin main

# ── จำลองเพื่อน: clone แล้วแก้/ push 2 รอบ ──

echo "Change 1 for file 1" >> file1.txt
git add . && git commit -m "First set of changes" && git push
echo "Change 2 for file 1" >> file1.txt
git add . && git commit -m "Second set of changes" && git push

# ── กลับมาที่ main-repo: fetch + ดู log + merge ──
cd ../main-repo
git fetch
git log --oneline
git log --all --oneline
git merge origin/main

สรุปคำสั่งสำคัญ (Cheat Sheet)

# ── เริ่มต้น repo ──────────────────────
git init                              # สร้าง repo ใหม่ในเครื่อง (เกิด .git)
git clone https://github.com/user/repo.git   # คัดลอก repo ทั้งก้อนลงมา

# ── ตรวจสอบ / ดูประวัติ ─────────────────
git status                            # ดูสถานะ working dir + staging
git log                               # ประวัติ commit เต็ม
git log --oneline                    # ประวัติแบบบรรทัดเดียว
git log --all --oneline              # ทุก branch
git diff                              # ดูความต่างที่ยังไม่ stage

# ── add & commit (Working → Staging → Repo) ──
git add file.txt                      # stage ไฟล์เดียว
git add .                             # stage ทุกไฟล์
git commit -m "message"             # บันทึก snapshot ลง repo

# ── remote (เชื่อม GitHub) ───────────────
git remote -v                         # ดู remote ที่ผูกไว้
git remote add origin https://url.git  # ผูก remote ชื่อ origin
git remote rename <old> <new>          # เปลี่ยนชื่อ remote
git remote remove <name>              # ลบ remote
git branch -M main                   # เปลี่ยนชื่อ branch ปัจจุบันเป็น main
git branch -r                         # ดู remote branches ที่รู้จัก

# ── push & pull & fetch ──────────────────
git push -u origin main               # push ครั้งแรก (ตั้ง upstream)
git push                              # push ครั้งต่อไป
git fetch                             # ดึงมาแต่ยังไม่ merge (ปลอดภัย)
git pull                              # = fetch + merge
git merge origin/main                 # รวม remote-tracking เข้า branch

# ── checkout (สำรวจ commit/branch) ───────
git checkout <commit-hash>           # ย้อนไปดู commit เก่า (detached HEAD)
git checkout main                     # กลับสู่ branch หลัก
🧭 จำเป็นแกน Week 2: add → commit → push (ขาขึ้น) และ fetch/pull (ขาลง) · จำสูตร git pull = git fetch + git merge ให้ขึ้นใจ
