🌲 Git Worktree

เปิดหลายสาขา (branch) พร้อมกันในหลายโฟลเดอร์ โดยใช้ repository เดียว — ทำงานคู่ขนานได้โดยไม่ต้อง stash หรือสลับ branch ไปมา

Git 2.5+ · เหมาะสำหรับงาน feature + hotfix พร้อมกัน

1 Git Worktree คืออะไร?

ปกติ repository หนึ่งตัวจะมีโฟลเดอร์ทำงาน (working directory) ได้เพียง หนึ่งเดียว และเช็คเอาท์ได้ทีละ branch เท่านั้น ถ้าอยากเปลี่ยน branch ต้องใช้ git checkout / git switch ซึ่งจะ "ทับ" ไฟล์ในโฟลเดอร์เดิมทั้งหมด

git worktree ให้เราสร้าง working directory เพิ่มได้หลายอัน โดยแต่ละอันเช็คเอาท์คนละ branch แต่ทั้งหมด ใช้ฐานข้อมูล Git (.git) ร่วมกัน — จึงไม่ต้อง clone ซ้ำ ไม่เปลืองพื้นที่ และ commit/branch เห็นถึงกันทุกโฟลเดอร์ทันที

2 ทำไมต้องใช้? (แก้ปัญหาอะไร)

ลองนึกภาพ: คุณกำลังเขียน feature ค้างอยู่ครึ่งทาง แล้วเจอบั๊กด่วนบน main ที่ต้องรีบแก้

😫 วิธีเดิม (stash + checkout)

  • git stash เก็บงานที่ทำค้าง
  • git checkout main สลับ branch
  • แก้บั๊ก · commit · push
  • git checkout feature กลับมา
  • git stash pop เอางานกลับ — เสี่ยง conflict
  • build/test ต้องรันใหม่ทุกครั้งที่สลับ

😎 ด้วย Worktree

  • งาน feature อยู่ในโฟลเดอร์เดิม ไม่ต้องแตะ
  • สร้าง worktree ใหม่จาก main ในอีกโฟลเดอร์
  • แก้บั๊กในโฟลเดอร์นั้น · commit · merge
  • กลับมาที่ feature — งานยังอยู่ครบเป๊ะ
  • ไม่มี stash · ไม่มี conflict · ไม่เสียงาน
  • เปิดคนละ terminal build คู่ขนานได้เลย
⚡

ไม่ต้องสลับ context

แต่ละ branch อยู่คนละโฟลเดอร์ ไม่ต้อง stash/unstash

🔀

ทำงานคู่ขนานจริง

build branch A พร้อมแก้โค้ด branch B ได้พร้อมกัน

💾

ประหยัดพื้นที่

ใช้ .git ร่วมกัน ไม่ต้อง clone ซ้ำหลายรอบ

🔍

รีวิว PR ง่าย

เปิด branch เพื่อนในโฟลเดอร์แยก โดยไม่กระทบงานตัวเอง

3 คำสั่งหลักที่ต้องรู้

คำสั่งทำอะไร
git worktree add <path> <branch>เช็คเอาท์ branch ที่มีอยู่แล้วไปไว้ในโฟลเดอร์ใหม่
git worktree add -b <new> <path>สร้าง branch ใหม่ + worktree พร้อมกันในคำสั่งเดียว
git worktree listแสดง worktree ทั้งหมด พร้อม branch ที่แต่ละอันเช็คเอาท์
git worktree remove <path>ลบ worktree ออกอย่างปลอดภัย (โฟลเดอร์ต้องสะอาด)
git worktree move <path> <new>ย้ายตำแหน่งโฟลเดอร์ worktree
git worktree pruneเก็บกวาดข้อมูลค้างของ worktree ที่ถูกลบโฟลเดอร์ทิ้งด้วยมือ
git worktree lock <path>ล็อกกัน prune (เช่น worktree อยู่บน external drive)

4 ตัวอย่างใช้งานจริง: Feature + Hotfix คู่ขนาน

1

เริ่มจาก repo หลัก

# สร้าง repo และ commit แรก
mkdir main-repo && cd main-repo
git init && git branch -M main
echo "console.log('v1');" > app.js
git add . && git commit -m "app v1"
2

สร้าง worktree ใหม่สำหรับ feature

# -b สร้าง branch ใหม่ + โฟลเดอร์ใหม่ในคราวเดียว
git worktree add -b feature-login ../feature-login main
git worktree list
3

ทำงาน feature ค้างไว้ (ยังไม่ commit)

cd ../feature-login
echo "function login(){}" > login.js
# กำลังเขียนค้าง... แล้วจู่ๆ เจอบั๊กด่วนบน main!
4

เปิด worktree สะอาดจาก main เพื่อแก้บั๊ก

git worktree add -b hotfix-typo ../hotfix-typo main
cd ../hotfix-typo
echo "console.log('v1 - fixed');" > app.js
git add app.js && git commit -m "hotfix"
5

merge hotfix แล้วกลับไปที่ feature

cd ../main-repo && git merge hotfix-typo
cd ../feature-login && git status
# ✅ login.js ที่เขียนค้างไว้ยังอยู่ครบ ไม่หายไปไหน!
💡 จุดสำคัญ: เราแก้บั๊กบน main และ merge เสร็จเรียบร้อย โดย ไม่ต้องแตะงาน feature ที่ค้างไว้เลย — ไม่มี stash ไม่เสี่ยงงานหาย

5 กฎสำคัญและข้อควรระวัง

⚠️ กฎ "หนึ่ง branch ต่อหนึ่ง worktree": branch หนึ่งจะถูกเช็คเอาท์ได้เพียง worktree เดียวเท่านั้น ถ้า main ถูกใช้ใน main-repo อยู่แล้ว การสั่ง git worktree add ../dup main จะถูกปฏิเสธ — เป็นการป้องกันไม่ให้ 2 โฟลเดอร์แก้ branch เดียวกันชนกัน
🛑 อย่าใช้ rm -rf ลบ worktree: ให้ใช้ git worktree remove <path> แทน เพราะ Git จะเก็บข้อมูล bookkeeping ไว้ ถ้าเผลอลบด้วยมือ ให้ตามเก็บกวาดด้วย git worktree prune
📌 ที่เก็บข้อมูล: worktree ย่อยจะมีไฟล์ .git เป็น ไฟล์ (ไม่ใช่โฟลเดอร์) ที่ชี้กลับไปยัง .git/worktrees/<name> ใน repo หลัก
